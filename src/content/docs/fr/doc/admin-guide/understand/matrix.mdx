Comment DNF implémente le serveur Matrix (Synapse) et les ponts mautrix.
Pour l'exploitation au quotidien, voir
[Matrix et les ponts](/fr/doc/admin-guide/operate/matrix/).

{/* t:p 0b2b60bf2deee753330eff128d4bfa6196323ac0a89a392ed8db957804a5dacf */}
## Vue d'ensemble

Tout est porté par `dnf/modules/service/matrix.nix`, avec deux modules
satellites&nbsp;: `element.nix` (client web statique) et `turn.nix` (relais
coturn pour l'audio/vidéo).

```d2
style.fill: transparent
direction: down

client: Clients\n(Element, Fractal) {
  shape: person
}
caddy: Caddy\n(reverse proxy)
synapse: Synapse {
  shape: rectangle
}
pg: PostgreSQL {
  shape: cylinder
}
idm: Kanidm\n(OIDC)
turn: coturn\n(TURN/STUN)

bridges: Ponts mautrix {
  wa: whatsapp
  sig: signal
  tg: telegram
  meta: messenger
  dc: discord
}

ext: Réseaux externes\n(WhatsApp, Telegram...) {
  shape: cloud
}

client -> caddy -> synapse -> pg
synapse -> idm
client -> turn
synapse <-> bridges <-> ext
```

- **Caddy** route `/_matrix/*` et `/_synapse/client/*` vers Synapse et sert
  les fichiers `/.well-known/matrix/{client,server}` (découverte des clients
  et fédération).
- **Synapse** écoute sur `dnfConfig.network.ports.matrix`, base PostgreSQL
  locale (créée avec collation `C` par une unité `matrix-db-init`).
- **Kanidm** fournit l'authentification&nbsp;: client OIDC provisionné via
  `darkone.service.idm.oauth2.matrix`, *local part* dérivée du
  `preferred_username` (voir [Authentification & IDM](/fr/doc/admin-guide/understand/auth-idm/)).
- **coturn** est branché automatiquement si le service `turn` est déclaré
  (`turn_uris`, secret partagé via sops).

## Anatomie du module

`matrix.nix` est un `lib.mkMerge` de blocs indépendants&nbsp;:

| Bloc | Condition | Contenu |
|---|---|---|
| Base | toujours | client OIDC, reverse proxy, persistance |
| Serveur | `matrix.enable` | Synapse, PostgreSQL, secrets communs, appservice `doublepuppet` |
| Un bloc par pont | `matrix.enable && bridges.<x>.enable` | secrets sops + service mautrix du pont |

Chaque pont est donc débrayable individuellement
(`darkone.service.matrix.bridges.{whatsapp,signal,telegram,messenger,discord}.enable`),
ses secrets sops n'étant déclarés que si le pont est actif.

## Les ponts mautrix

DNF s'appuie sur les modules nixpkgs `services.mautrix-*`, qui génèrent la
*registration* appservice et l'enregistrent auprès de Synapse au démarrage.
Deux générations de ponts cohabitent, avec des clés de configuration
différentes&nbsp;:

| Pont | Génération | Double puppeting | Niveau |
|---|---|---|---|
| whatsapp | bridgev2 (Go) | `double_puppet.secrets` | `user` |
| signal | bridgev2 (Go) | `double_puppet.secrets` | `user` |
| messenger (meta) | bridgev2 (Go) | `double_puppet.secrets` | `user` |
| telegram | legacy (Python) | `bridge.login_shared_secret_map` | `full` |
| discord | legacy (Go) | `bridge.login_shared_secret_map` | `user` |

<Aside type="note" title="Pourquoi « full » pour Telegram ?">
Sur le pont Telegram legacy, le niveau `user` ne permet pas de connecter son
propre compte (il faut `puppeting` au minimum). DNF donne `full` au domaine
pour que chaque compte local puisse se connecter et partager ses salons.
</Aside>

{/* t:p 872be0f0382f211b1da743be9b7ac2166fabde666dfde51b9691a7ae76e2a110 */}
### Permissions

Le helper `mkBridgePermissions` (dans `matrix.nix`) construit la même
politique pour tous les ponts&nbsp;:

```yaml
permissions:
  "domain.tld": user          # tout compte local, avec son propre compte distant
  "@alice:domain.tld": admin  # le matrix.admin déclaré dans etc/config.yaml
```

Les sessions distantes sont isolées par utilisateur Matrix&nbsp;: la
généralisation à tous les comptes du domaine est sans risque d'interférence.

{/* t:p 580fec2b6739ee349a69326fb1515efd838f21ba016a7e3767969df39b631af8 */}
### Double puppeting

:::note[A quoi ça sert ?]
Sans double puppeting, les messages relayés par un pont apparaissent dans
Matrix comme envoyés par le **robot** du pont (par ex. `@whatsappbot:domain.tld`).
Avec le double puppeting, le pont utilise le **compte Matrix de l'utilisateur**
pour envoyer les messages&nbsp;: le salon affiche «&nbsp;Alice a envoyé…&nbsp;» avec sa
vraie identité, comme dans une conversation Matrix normale.

C'est transparent pour l'utilisateur, qui voit ses messages (entrants et sortants) venir
de son propre compte, sans savoir qu'ils transitent par un pont.
:::

Méthode officielle [appservice](https://docs.mau.fi/bridges/general/double-puppeting.html)&nbsp;:
un appservice `doublepuppet` est enregistré dans Synapse
(`app_service_config_files`) avec un namespace couvrant `@.*:domain.tld`.
Son `as_token` autorise chaque pont à émettre des événements **au nom du
vrai compte** de l'utilisateur.

```d2
style.fill: transparent
direction: down

phone: Téléphone\n(WhatsApp) {
  shape: person
}
bridge: Pont mautrix
synapse: Synapse
room: Salon Matrix\n(message de @alice) {
  shape: page
}

phone -> bridge: message sortant
bridge -> synapse: as_token doublepuppet
synapse -> room
```

Le token est unique pour tous les ponts (`mautrix-doublepuppet-as-token`) et
injecté dans chaque config par substitution de variables d'environnement
(`as_token:$MAUTRIX_DOUBLEPUPPET_AS_TOKEN`), les modules nixpkgs passant la
configuration par `envsubst` avec le `environmentFile` du service.

### Les tokens d'appservice

La *registration* d'un appservice est le contrat entre Synapse et le pont.
Elle contient deux tokens, un par sens d'authentification&nbsp;:

| Token | Sens | Usage |
|---|---|---|
| `as_token` | pont → Synapse | le pont s'authentifie sur l'API client (envoi de messages, sync...) |
| `hs_token` | Synapse → pont | Synapse s'authentifie quand il pousse les événements au pont (`/transactions`) |

Par défaut, les modules nixpkgs **génèrent ces tokens aléatoirement** au
premier démarrage et les stockent dans `/var/lib/mautrix-*`. Conséquence&nbsp;:
toute réinitialisation du répertoire d'état change la paire de tokens, alors
que Synapse — qui ne lit les registrations **qu'au démarrage** — garde
l'ancienne en mémoire&nbsp;; le pont est alors rejeté
(«&nbsp;The as_token was not accepted&nbsp;»).

DNF fixe donc les deux tokens de chaque pont via sops&nbsp;: la registration
devient une **fonction pure des secrets**. Réinitialiser l'état d'un pont
n'invalide plus rien côté Synapse, et l'ensemble est reproductible
(redéploiement, migration, restauration de sauvegarde).

<Aside type="note" title="Le cas doublepuppet">
L'appservice `doublepuppet` a une `url` vide&nbsp;: Synapse ne lui pousse
jamais rien, son `hs_token` n'est donc jamais utilisé — il n'est requis que
par le format de la registration.
</Aside>

## Fédération

La fédération Matrix est la communication **serveur à serveur**&nbsp;: un compte
`@alice:domain.tld` dialogue avec `@bob:autre.org` sans compte sur l'autre
serveur. DNF la rend **configurable et explicite** via
`darkone.service.matrix.federation`.

{/* t:p b23cdfc68daffeebba0a53cf27f81128ea5359de0d7287fec23fd6fb18533020 */}
### Découverte ≠ connectivité

Deux notions distinctes, souvent confondues&nbsp;:

- **Découverte**&nbsp;: apparaître dans les annuaires distants (recherche de
  salons, de profils). DNF la **verrouille par défaut**
  (`allow_public_rooms_over_federation = false`, annuaire utilisateurs
  désactivé, profils privés sur la fédération). Le réseau reste **joignable mais
  non cherchable**.
- **Connectivité**&nbsp;: la capacité d'échanger des événements avec un autre
  serveur. C'est ce que pilote la fédération.

<Aside type="caution" title="Masquer n'est pas cacher">
Retirer un salon des annuaires ne le rend pas secret&nbsp;: qui connaît son
alias (`#salon:domain.tld`) peut tenter de le rejoindre. Pour un salon
réellement privé&nbsp;: non listé **et** en accès `knock` ou sur invitation.
</Aside>

{/* t:p 765c6b4ee94af9bed3144c24c1918bed068f05671750f24f38c55342bac71794 */}
### Trois modes

Le module traduit deux options en une seule clé Synapse
`federation_domain_whitelist`, dont la **présence** change le sens&nbsp;:

| Réglage | Clé Synapse | Effet |
|---|---|---|
| `enable = false` | `[ ]` (liste vide) | fédération **totalement bloquée** |
| `enable = true; whitelist = [ ]` | *(absente)* | fédération **ouverte** à tous |
| `enable = true; whitelist = [ "ami.org" ]` | `[ "ami.org" ]` | **liste blanche** (entrant + sortant) |

La liste blanche est **bidirectionnelle**&nbsp;: hors des domaines listés, ni
entrant ni sortant. C'est le réglage recommandé pour un réseau familial. La
délégation (`/.well-known/matrix/server` → `matrix.domain.tld:443`) étant servie
par Caddy, aucun port 8448 dédié n'est nécessaire.

## Comptes&nbsp;: SSO et amis

Deux origines de comptes cohabitent sur le même serveur&nbsp;:

- **Utilisateurs déclarés**&nbsp;: authentifiés par le SSO Kanidm (OIDC). Aucun
  mot de passe Matrix, aucune création de compte (voir
  [Authentification & IDM](/fr/doc/admin-guide/understand/auth-idm/)).
- **Amis**&nbsp;: comptes locaux à mot de passe, ouverts par inscription **sur
  jeton** (`friendRegistration.enable`). Sans jeton, aucune inscription possible.

Les deux types partagent les mêmes droits (ponts inclus). La création d'un jeton
exige un *server admin* (voir
[le guide d'exploitation](/fr/doc/admin-guide/operate/matrix/#inscrire-des-amis-comptes-locaux)).

## Authentification nouvelle génération (MAS)

Optionnel (`darkone.service.matrix.mas.enable`), le
[Matrix Authentication Service](https://element-hq.github.io/matrix-authentication-service/)
(MAS) déporte **toute** l'authentification hors de Synapse&nbsp;: connexion,
comptes, sessions, portail self-service `/account`. C'est le socle
d'[Element X](https://element.io/app/) (connexion par QR code) et la voie
officielle, l'authentification interne de Synapse étant en fin de vie.

```d2
style.fill: transparent
direction: down

client: Clients\n(Element, Element X) {
  shape: person
}
caddy: Caddy\n(matrix.domain.tld)
mas: MAS\n(authentification, comptes)
synapse: Synapse
idm: Kanidm\n(OIDC)

client -> caddy
caddy -> synapse: "/_matrix/*, /_synapse/*"
caddy -> mas: "/, login, logout, refresh"
synapse <-> mas: "introspection, provisioning"
mas -> idm: SSO amont
```

- **Un seul vhost**&nbsp;: MAS partage `matrix.domain.tld`. Caddy lui route la
  racine (pages de connexion, `/account`, `/oauth2/*`) et les trois endpoints
  de compatibilité (`login`, `logout`, `refresh`)&nbsp;; Synapse garde
  `/_matrix/*` et `/_synapse/*`. Aucun sous-domaine ni DNS supplémentaire.
- **Kanidm reste la source d'identité**&nbsp;: MAS devient le client OIDC à la
  place de Synapse (mêmes identifiant et secret), déclaré comme fournisseur
  amont sous un ULID fixe. Synapse ne fait plus que demander à MAS de valider
  chaque token (secret partagé, loopback).
- **Découverte automatique**&nbsp;: relié à MAS, Synapse sert lui-même
  `auth_metadata` ([MSC2965](https://github.com/matrix-org/matrix-spec-proposals/pull/2965)).
  Les clients trouvent MAS sans modification du `.well-known`.
- **Ponts inchangés**&nbsp;: registrations et double puppeting fonctionnent à
  l'identique. Seule adaptation, automatique&nbsp;: les ponts gèrent leurs
  devices e2ee via
  [MSC4190](https://github.com/matrix-org/matrix-spec-proposals/pull/4190),
  l'API `/login` d'appservice n'existant plus.
- **Amis**&nbsp;: l'inscription sur jeton bascule de Synapse vers MAS
  (`friendRegistration.enable` reste le commutateur)&nbsp;; les comptes à mot
  de passe continuent de fonctionner via la couche de compatibilité.

Activation et limitations&nbsp;: voir
[le guide d'exploitation](/fr/doc/admin-guide/operate/matrix/#authentification-nouvelle-génération-mas).

## Secrets et ports

- **Secrets** : déclarés par bloc dans `matrix.nix` (`sops.secrets.*`), et
  assemblés en fichiers d'environnement par des `sops.templates.*` possédés
  par l'utilisateur système du pont. Liste complète dans
  [la page d'exploitation](/fr/doc/admin-guide/operate/matrix/#secrets-requis).
- **Ports** : `matrix`, `matrixTelegram` et `matrixDiscord` sont fixés par le
  registre `dnf/config/network.nix`&nbsp;; les ports appservice par défaut des
  autres ponts (29318, 29319, 29328) y sont listés en `reserved` pour éviter
  toute collision future.
- **Statuts WhatsApp** : `network.enable_status_broadcast = false` dans la
  config du pont, sinon le bridge recrée indéfiniment le salon
  « WhatsApp Status Broadcast » pour chaque utilisateur.
