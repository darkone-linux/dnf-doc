Le parc se déploie avec [colmena](https://github.com/zhaofengli/colmena) : une
commande construit et applique la configuration sur un ou plusieurs hôtes, à
distance, depuis le poste admin.

{/* t:p d4a06fc139c0d5029ddc5205311c422d5da5a5e1fe1b2e6d014d42aae5b8629c */}
## Appliquer une configuration

```sh
just apply <cible> [action]    # alias : a
```

- **cible** = nom d'hôte, motif (`'*'`), liste (`a,b`) ou tag colmena (`@server`).
- **action** = `switch` (défaut), `boot`, `test` ou `build`.

| Commande | Pour quoi |
|---|---|
| `just apply <cible>` | Construire + activer sur la (les) cible(s) |
| `just apply-local` | Appliquer sur la machine courante (alias `al`) |
| `just apply-verbose <cible>` | Idem `apply` en mode trace (alias `av`) |

:::tip[Construire sur la cible]
`apply` construit **sur l'hôte cible** (`--build-on-target`) : le poste admin n'a
pas besoin de la puissance ni de l'architecture des machines déployées.
:::

## Déployer en sécurité

**Monter en confiance par l'action** : chaque étape est moins risquée que la
suivante.

```sh
just apply <hôte> build      # télécharge + compile : 100 % sûr, rien n'est activé
just apply <hôte> test       # active sans switcher : ni génération, ni boot
just apply <hôte>            # switch : active et crée une nouvelle génération
```

**Progresser du cœur vers la périphérie** : déployez les nœuds dans cet ordre,
pour ne jamais vous couper l'accès à un nœud par celui qui le précède.

```d2
style.fill: transparent
direction: right

hcs: HCS
gwl: Passerelle locale
hl: Hôtes locaux
gwd: Passerelles distantes
hd: Hôtes distants

hcs -> gwl -> hl -> gwd -> hd
```

:::caution[Pourquoi cet ordre]
La connectivité d'un hôte distant dépend de sa passerelle, qui dépend du VPN
coordonné par le HCS. Déployer un maillon avant son support peut rendre les
suivants injoignables.
:::

## Mettre à jour

Le système est figé par les *flakes*. Mettre à jour = rafraîchir les entrées,
puis redéployer.

```sh
just update-flake     # met à jour dnf/ + racine, commit les locks
just apply '*'        # déploie la mise à jour
```

:::caution[Toujours tester avant]
Avant un déploiement large, validez localement : `just check-flake` (ou
`just check-all`), puis `just apply <hôte-pilote>` sur **un** hôte témoin.
:::

## Cache binaire

Deux services se combinent pour éviter de recompiler ou de retélécharger les
dérivations à chaque déploiement.

| Service | Rôle |
|---|---|
| `harmonia` | Sert le `/nix/store` local, signé, **directement** aux hôtes (LAN, ou VPN s'il est `global`). |
| `nix-cache` | Proxy `nginx` par zone qui met en cache le cache public `cache.nixos.org` sur la passerelle. |

Un hôte récupère donc une dérivation dans cet ordre&nbsp;: la `harmonia` de sa
zone, une éventuelle `harmonia` `global`, puis le proxy `nix-cache` de sa zone.
La première récupération depuis Internet est ainsi mutualisée pour toute la zone.

:::tip[Hôte nomade (portable multi-zones)]
Un portable qui change de zone ne doit pas garder des substituters figés sur sa
zone déclarée&nbsp;: injoignables ailleurs, ils coûtent des timeouts à chaque
requête. Activez le mode *roaming*&nbsp;:

```nix
# usr/machines/<hôte>/default.nix
darkone.service.nix-cache.roaming = true;
```

Ses substituters deviennent des noms **neutres** (`harmonia.dnf.internal`,
`nix-cache.dnf.internal`) que le DNS de chaque zone résout vers ses propres
caches → le cache suit la machine. Hors de toute zone, réponse NXDOMAIN
immédiate → repli direct sur `cache.nixos.org`, sans délai. Les clés de
signature étant communes à tout le réseau, rien d'autre à configurer.
:::

:::note[Activation]
`harmonia` et `nix-cache` sont des [services](/fr/doc/admin-guide/operate/services/)
à activer sur un hôte du réseau (`nix-cache` sur la passerelle de zone). La clé
de signature de harmonia est gérée par
[sops](/fr/doc/admin-guide/operate/secrets/) (`just configure-admin-host`).
:::

:::tip[Pourquoi un proxy à upstream unique]
`nix-cache` ne relaie qu'**un seul** upstream (`cache.nixos.org`)&nbsp;: il met
en cache `narinfo` et `nar` tels quels, signatures intactes. Pas de réécriture
d'URL ni de base de données, donc pas l'erreur HTTP 500 « invalid nar hash » de
l'ancien relais `ncps`.
:::
