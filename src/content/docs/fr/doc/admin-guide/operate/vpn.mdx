Le VPN maille toutes les zones et les machines distantes :
[Headscale](https://headscale.net/) coordonne (sur le HCS),
[Tailscale](https://tailscale.com/) est le client sur chaque nœud. Vue
d'ensemble dans [Réseau](/fr/doc/admin-guide/understand/network/#réseau-étendu-vpn).

{/* t:p be556da93b9b830aa8e89f43b69dd1d5eecc77324080157f7f93e3fa42167c45 */}
## Activer le VPN

Activer la coordination et déclarer le HCS (`profile: hcs`) suffit :

```yaml
# etc/config.yaml
network:
  coordination:
    enable: true
    hostname: "hcs"
    domain: "headscale"
```

Les rôles se déduisent du profil de chaque hôte :

- **HCS** : coordination du maillage (Headscale).
- **Passerelle** : *subnet router* + *exit node* : publie le sous-réseau de sa zone.
- **Autres nœuds** : clients Tailscale.

## Préparer le tailnet (une fois)

Toutes les machines rejoignent le maillage via un **utilisateur Headscale
d'attache** partagé (`nix`). Il se crée **une seule fois** pour tout le réseau ;
s'il existe déjà, passer cette étape.

```sh
just enter hcs
sudo headscale users list        # déjà présent ? → rien à faire
sudo headscale users create nix --display-name "Nix Admin" --email "nix@domain.tld"
```

:::caution[Non modifiable]
L'e-mail et le nom affiché d'un utilisateur ne sont **pas** modifiables après création.
:::

## Enregistrer une passerelle

Le profil `gateway` fait déjà de la passerelle un **client Tailscale** : au
déploiement, le service `tailscaled-autoconnect` l'enregistre tout seul sur
Headscale et annonce le sous-réseau de sa zone. Il reste deux gestes manuels :
fournir une **clé d'attache** dans sops (avant le déploiement) et **approuver les
routes** sur le HCS (après).

<Steps>

1. **Sur le HCS : générer une clé d'attache**

   ```sh
   just enter hcs
   sudo headscale users list                                    # ID de l'utilisateur nix
   sudo headscale preauthkeys create --reusable --expiration 1h --user <id>
   ```

   Copier la clé affichée (`tskey-...`). Elle n'a besoin de vivre que le temps du
   déploiement.

2. **Déposer la clé dans les secrets**

   ```sh
   just sops        # ouvre usr/secrets/secrets.yaml
   ```

   Renseigner la clé sous `tailscale/authKey` :

   ```yaml
   # usr/secrets/secrets.yaml
   tailscale:
       authKey: "tskey-..."
   ```

   :::caution[Avant l'installation]
   Poser cette clé **avant** le premier `just apply` (ou `just full-install`) de la
   passerelle : sans le secret `tailscale/authKey`, l'activation sops échoue.
   :::

3. **Déployer la passerelle**

   ```sh
   just apply gw
   ```

   La passerelle s'enregistre automatiquement. En interne, le démon exécute :

   ```sh
   tailscale up \
     --login-server https://headscale.domain.tld \
     --auth-key file:/run/secrets/tailscale/authKey \
     --advertise-routes 10.0.0.0/16 --accept-routes --reset
   ```

   :::note[Rien à taper à la main]
   Inutile de lancer `tailscale up` ou `tailscale set` soi-même : le service
   `tailscaled-autoconnect` applique ces options (login-server, clé, routes) à
   chaque démarrage. Si l'enregistrement n'a pas eu lieu (HCS injoignable au
   déploiement), le [watchdog](#auto-réparation-du-client) le rejoue&nbsp;; on peut
   aussi forcer avec `sudo systemctl restart tailscaled-autoconnect`.
   :::

4. **Sur le HCS : approuver les routes annoncées**

   ```sh
   sudo headscale nodes list                                    # ID de la passerelle
   sudo headscale nodes approve-routes --identifier <id> --routes 10.0.0.0/16
   ```

5. **Déclarer l'IP tailnet de la passerelle**

   À l'enregistrement, Headscale attribue à la passerelle une IP tailnet
   (`100.64.x.y`), visible dans `headscale nodes list`. La reporter dans
   `etc/config.yaml`&nbsp;: les autres nœuds joignent la zone (DNS interne,
   réplication, supervision) par cette adresse.

   ```yaml
   # etc/config.yaml
   zones:
     maison:
       gateway:
         vpn:
           ipv4: "100.64.0.8"
   ```

   Puis régénérer et redéployer les nœuds concernés (le HCS et les clients de la
   zone) pour propager l'adresse.

</Steps>

## Administration

La commande `headscale` (alias `h` = `sudo headscale`) sur le HCS :

```sh
sudo headscale users list      # utilisateurs
sudo headscale nodes list      # clients connectés
sudo headscale nodes routes    # routes annoncées / approuvées
```

## Auto-réparation du client

Un `tailscaled` peut **décrocher silencieusement** de Headscale : la connexion de
contrôle est perdue alors que le service reste « actif », et le sous-réseau d'une
passerelle devient injoignable jusqu'à un redémarrage manuel. Un *watchdog* local
détecte et corrige ce cas, sur **chaque client** Tailscale.

| Aspect | Comportement |
|---|---|
| Sonde | toutes les 60&nbsp;s : `tailscale status` (backend, état en ligne, santé) |
| Anti-rebond | 3 sondes en échec d'affilée (~3&nbsp;min) avant d'agir : encaisse les coupures WAN passagères |
| Correction | `systemctl restart tailscaled` (ré-enregistrement automatique sur Headscale) |
| Anti-boucle | une seule correction par fenêtre de 10&nbsp;min |
| Observabilité | métriques `dnf_tailscale_healthy` et `dnf_tailscale_selfheal_restarts_total` (textfile node_exporter) |

Deux [alertes](/fr/doc/admin-guide/operate/alerts/) accompagnent le mécanisme :

- **`TailscaleUnhealthy`** : toujours décroché malgré la correction (problème de fond) ;
- **`TailscaleFlapping`** : l'auto-réparation se répète trop souvent (cause à traiter).

:::note[Activé par défaut]
Le watchdog tourne sur tout client Tailscale. Pour le désactiver sur un hôte :
`darkone.service.tailscale.selfHeal.enable = false;`.
:::
