Les commandes ci-dessous sont toutes exécutées avec [`just`](https://just.systems/man/).

```sh title=Utilisation
just <commande> <arguments>
```

Chaque commande est préfixée d'une icône indiquant son contexte d'utilisation :

- 💠 **Commun** : disponible dans le projet comme dans le framework.
- 🔆 **Framework** : spécifique au framework DNF amont (`dnf/`).
- ✳️ **Projet** : spécifique au projet consommateur (mon réseau).

:::caution[Élévation des privilèges]
L'essentiel des commandes s'exécute avec l'utilisateur qui administre le projet, mais
certaines d'entre elles élèvent les privilèges sur l'utilisateur `nix` (`sudo -u nix ...`).
:::

:::tip[Commandes clés]
Les commandes marquées d'une ⭐ sont les plus utiles au quotidien.
:::

## Commandes GIT

Ces commandes travaillent sur **tous les dépôts** du projet (par défaut
`src/*/`, `dnf`, `doc`, et la racine). L'alias court est `g`.

{/* t:p a1e6bad8fb415d7340e9d07849875119abef92d8dc270e84321e520836107748 */}
### ✳️ `git <cmd> [projet]`

Exécute n'importe quelle commande git sur un ou tous les projets.

:::tip[Projets itérés]
Listés par `just git-show-repos` (par défaut tous).
:::

```sh
just git "log --oneline -5"       # derniers commits de tous projets
just git "status" dnf             # git status sur dnf/ seulement
just git "remote -v"              # remotes de tous les projets
```

{/* t:p 48edf57ea025e134f58bc136457844f56ed64d9df6c39185730c9bd89dd2a4c8 */}
### ✳️ `commit <message> [projet]` ⭐

`add -A` + `commit -m` sur un ou tous les projets. Si le message est
identique au dernier commit, la recette **amende** au lieu de créer un nouveau
commit.

```sh
just commit "fix(net): update dns config"
just commit "fix(net): update dns config" dnf    # dnf/ seulement
```

{/* t:p 90027dc80a9496cb27b392290b13a3820287445362d9d2e490deb6beaac7c348 */}

### ✳️ `update-flake`

Met à jour les entrées flake de `dnf/` puis de la racine. Si le dernier commit
de `dnf/` est déjà un `update(flake):`, il l'amende.

:::caution[Ne pas faire en production]
`nix flake update` monte toutes les entrées à leur dernière version : à
n'utiliser que pour une mise à jour régulière et testée.
:::

```sh
just update-flake
```

{/* t:p d9a3d2b8f7d76bab8b26a3a01b5ff12c131b35343920df057d94b8e965d5b26d */}
### ✳️ `update-dnf`

Synchronise `flake.lock` avec le HEAD local de `dnf/` (co-développement).

:::note[Aucun checkout dnf/]
Si `dnf/.git` n'existe pas (mode release), la commande ne fait rien.
:::

```sh
just update-dnf
```

## Commandes d'installation

L'installation d'un nouvel hôte suit un enchaînement précis.

{/* t:p d14baec5b0376679bae1d0a0a003046361405791774a8844fdc1752ef074921c */}
### ✳️ `install <hôte> [user] [ip]`

Formate les disques via **disko** et installe NixOS via **nixos-anywhere**.

:::danger[Destructeur]
Toutes les données sur l'hôte cible sont effacées. Une confirmation est
demandée avant d'exécuter.
:::

:::note[Configuration requise]
L'hôte doit avoir une configuration disko dans `etc/config.yaml` et un
dossier `usr/machines/<hôte>/default.nix`.
:::

```sh
just install myserver
just install myserver nixos 10.0.0.5     # IP personnalisée
just install myserver do=test             # VM test uniquement
```

{/* t:p ac3b2fd2d96f8a080081e401177db29d977c3da3fc6a0fcfd8cfd58df4d1ad33 */}

Chaîne complète : `install` + attente reboot + `configure` + `apply` + `gc` + reboot.

```sh

```

### ✳️ `luks <hôte> [action] [ip]`

Provisionne le chiffrement d'un hôte : passphrases partagée et par hôte dans
sops, manifeste `usr/secrets/luks.json`, clé SSH de l'initrd sur la cible,
IP WAN des passerelles. `action=passwd` fait tourner la passphrase par hôte.
Voir [Chiffrement des disques (LUKS)](/fr/doc/admin-guide/operate/luks/).

:::tip[Idempotent]
Appelée par `just configure` ; sans volume LUKS dans le disko de l'hôte,
c'est un no-op journalisé.
:::

```sh
just luks myserver
just luks myserver passwd
```

{/* t:p 68a3a1962d58418a2c6ca2087c01b8a6f09b61aede53766186715eebd7c0eff9 */}
### ✳️ `apply <cible> [action]` / `a` ⭐

Déploie la configuration sur une ou plusieurs cibles.

| Paramètre | Description |
|---|---|
| `cible` | Nom d'hôte, motif (`'*'`), liste (`a,b`), tag colmena (`@server`) |
| `action` | `switch` (défaut), `boot`, `test`, `build` |

:::note[Construction sur la cible]
`--build-on-target` : la compilation a lieu sur l'hôte déployé, pas sur le poste admin.
:::

```sh
just apply @all                 # tous les hôtes
just apply myserver             # switch (défaut)
just apply myserver boot        # boot
just apply 'db-*'               # tous les hôtes commençant par db-
```

### ✳️ `yubikey <utilisateur> [clé] [action]`

Enrôle une YubiKey (`main` par défaut, ou `backup`) pour un utilisateur : PAM
(session/sudo), LUKS et lien d'enrôlement Kanidm, en un seul passage sur le
poste admin. `action=revoke` révoque la clé sur tout le parc au prochain
déploiement. Voir [Authentification forte](/fr/doc/admin-guide/operate/yubikey/).

```sh
just yubikey darkone            # enrôle la clé principale
just yubikey darkone backup     # enrôle la clé de secours
just yubikey darkone main revoke
```

### ✳️ `rotate init|push-keys|finalize`

Rotation de la clé age `/etc/sops/age/infra.key` en 3 étapes avec vérifications
d'intégrité.

| Étape | Action |
|---|---|
| `init` | Génère une nouvelle clé, conserve l'ancienne, rechiffre les secrets |
| `push-keys` | Pousse la clé combinée (nouvelle + ancienne) vers tous les hôtes |
| `finalize` | Vérifie que tous les hôtes ont la nouvelle clé, supprime l'ancienne |

:::danger[Rotation non terminée]
Ne pas interrompre une rotation. Les sidecars `.cur` / `.old` marquent une rotation en cours.
:::

```sh
just rotate init
just rotate push-keys
just rotate finalize
```

## Commandes avancées

Ces commandes couvrent des fonctionnalités spécifiques : provisionnement du bot
d'alertes Matrix et gestion de la réplication Kanidm.

{/* t:p 6f5392cdf438fd8eefc76271cb14d343baf4a5593495236de95b8b35b970a42f */}
### ✳️ `configure-alert-bot`

Crée ou rafraîchit le bot Matrix, le secret webhook et les salons d'alertes
et d'incidents depuis la section `network.matrix` de `config.yaml`.

```sh
just configure-alert-bot
```

{/* t:p c42e83fd9c7a4f3c783eedfafd3f9f0b7516069ed1a976f4f14797c206d917bf */}
### ✳️ `idm-sync-certs`

Collecte les certificats de réplication Kanidm de tous les nœuds du parc
vers `usr/secrets/replication/`.

:::note[Déploiement en deux temps]
1. `just apply` : chaque nœud génère son identité.
2. `just idm-sync-certs` : collecte les certificats publics.
3. Commit + `just apply` : la réplication démarre.
:::

```sh
just idm-sync-certs
```

{/* t:p a5060e0adb11a791d702913bb1c2058a9b3312faa5a700f35dad0dc8c5f994b8 */}
### ✳️ `idm-fix-replica <suppl> <cons>`

Analyse la santé de la réplication Kanidm sans rien modifier. Prend le nœud
supplier (HCS) et la liste des consumers (gateways, séparés par des virgules).

```sh
just idm-fix-replica hcs agate,gw
```
