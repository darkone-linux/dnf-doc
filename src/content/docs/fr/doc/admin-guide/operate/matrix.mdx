Activer le serveur Matrix (Synapse), son client web Element et les ponts
mautrix (WhatsApp, Signal, Telegram, Messenger, Discord). Pour l'architecture, voir
[Matrix dans DNF](/fr/doc/admin-guide/understand/matrix/).

## Activer Matrix

<Steps>

1. Déclarer les services sur l'hôte cible dans `etc/config.yaml`&nbsp;:

   ```yaml
   # etc/config.yaml
       services:
         matrix:
           title: "Matrix"
           description: "Messagerie instantanée"
         element:        # client web
         turn:           # relais audio/vidéo (recommandé)
   ```

2. Déclarer l'administrateur Matrix (clé globale, hors hôtes)&nbsp;:

   ```yaml
   # etc/config.yaml
   matrix:
     admin: "alice"    # local part du compte administrateur
   ```

   Ce compte administre les ponts et reçoit les
   [alertes de supervision](/fr/doc/admin-guide/operate/alerts/).

3. Créer les [secrets](/fr/doc/admin-guide/operate/secrets/) requis
   (voir tableau ci-dessous), puis déployer.

</Steps>

L'authentification passe par le [SSO Kanidm](/fr/doc/admin-guide/operate/sso/)
(OIDC)&nbsp;: tout utilisateur déclaré se connecte avec son compte réseau,
aucune création de compte Matrix n'est nécessaire.

{/* t:p 612e4b4b4c6fdc9f568eccb61bb4176b608d5b885eb2f88cbbb765f0d7e6f941 */}
### Secrets requis

À renseigner dans `usr/secrets/secrets.yaml`&nbsp;:

| Clé sops | Rôle | Génération |
|---|---|---|
| `matrix-rss-password` | Registration shared secret Synapse | `openssl rand -hex 32` |
| `oidc-secret-matrix` | Secret du client OIDC Kanidm | cf. [SSO](/fr/doc/admin-guide/operate/sso/) |
| `turn-secret` | Secret partagé coturn (si `turn` activé) | `openssl rand -hex 32` |
| `mautrix-doublepuppet-as-token` | Token appservice *double puppeting* (commun aux ponts) | `openssl rand -hex 32` |
| `mautrix-doublepuppet-hs-token` | Idem, requis par Synapse, jamais utilisé | `openssl rand -hex 32` |

## Gérer les ponts

Les ponts WhatsApp, Signal, Telegram et Messenger sont **activés par
défaut** avec Matrix&nbsp;; Discord est **désactivé par défaut**. Réglage par hôte dans la
configuration nix de la machine&nbsp;:

```nix
# usr/machines/<host>/default.nix
darkone.service.matrix.bridges = {
  discord.enable = true;     # opt-in
  messenger.enable = false;  # désactiver un pont par défaut
};
```

Chaque pont a ses secrets propres&nbsp;:

| Pont | Clés sops | Remarque |
|---|---|---|
| WhatsApp | `mautrix-whatsapp-as-token`, `mautrix-whatsapp-hs-token`, `mautrix-whatsapp-encryption-pickle-key` | `openssl rand -hex 32` chacun |
| Signal | `mautrix-signal-as-token`, `mautrix-signal-hs-token`, `mautrix-signal-encryption-pickle-key` | `openssl rand -hex 32` chacun |
| Messenger | `mautrix-meta-as-token`, `mautrix-meta-hs-token`, `mautrix-meta-encryption-pickle-key` | `openssl rand -hex 32` chacun |
| Telegram | `mautrix-telegram-api-id`, `mautrix-telegram-api-hash`, `mautrix-telegram-as-token`, `mautrix-telegram-hs-token` | API id/hash à créer sur [my.telegram.org](https://my.telegram.org) |
| Discord | `mautrix-discord-as-token`, `mautrix-discord-hs-token` | `openssl rand -hex 32` chacun |

Les tokens `as`/`hs` rendent la *registration* appservice de chaque pont
déterministe (voir
[les tokens d'appservice](/fr/doc/admin-guide/understand/matrix/#les-tokens-dappservice)).

<Aside type="caution" title="Migration d'un pont déjà déployé">
Sur un pont WhatsApp ou Signal déployé **avant** l'introduction de ces tokens,
l'enregistrement existant conserve les anciens tokens auto-générés. Pour basculer
sur les tokens sops&nbsp;: supprimer `/var/lib/mautrix-<pont>/<pont>-registration.yaml`,
redémarrer le pont (l'enregistrement est régénérée depuis la config), puis
redémarrer `matrix-synapse`. Pour Messenger et Discord, un simple redéploiement
suivi d'un redémarrage de Synapse suffit&nbsp;: leurs modules réécrivent la
registration depuis la config à chaque démarrage.
</Aside>

### Permissions

Rien à gérer&nbsp;: tout compte local (`@*:domain.tld`) peut utiliser chaque
pont avec **son propre compte** distant, sans interférence entre
utilisateurs. Le compte `matrix.admin` dispose des commandes
d'administration des bots (`help` en conversation directe avec le bot pour
la liste).

{/* t:p 57866a0b0c836e99e3a85df4799b3a07fa81c8c06bcd8807ab57b6fae2e5fbb9 */}
### Double puppeting

Les ponts partagent un appservice `doublepuppet` (token
`mautrix-doublepuppet-as-token`)&nbsp;: les messages envoyés depuis le
téléphone apparaissent dans Matrix comme venant du **vrai compte** de
l'utilisateur. C'est automatique pour tous les comptes locaux, aucune action
par utilisateur.

## Ouvrir le réseau à la fédération

Par défaut, le serveur fédère avec **tous** les serveurs Matrix, tout en restant
**invisible dans les recherches** (annuaires non exposés). Pour contrôler la
portée, régler `darkone.service.matrix.federation` dans la config nix de
l'hôte&nbsp;:

```nix
# usr/machines/<host>/default.nix
darkone.service.matrix.federation = {
  enable = true;       # false = fédération totalement bloquée
  whitelist = [ ];     # vide = ouverte ; sinon liste blanche
};
```

| Objectif | Réglage |
|---|---|
| Réseau isolé (aucune fédération) | `enable = false;` |
| Ouvert à tout l'écosystème Matrix | `enable = true; whitelist = [ ];` |
| Restreint à des serveurs de confiance | `enable = true; whitelist = [ "ami.org" "matrix.org" ];` |

<Aside type="caution" title="Réseau familial">
La liste blanche est **bidirectionnelle**&nbsp;: un serveur non listé ne peut ni
écrire à vos utilisateurs ni recevoir d'eux. Pour un réseau utilisé par des
enfants, renseigner `whitelist` plutôt que de laisser la fédération ouverte.
</Aside>

Vérifier après déploiement&nbsp;:

<Steps>

1. Tester la fédération entrante avec
   [federationtester.matrix.org](https://federationtester.matrix.org/) sur
   `domain.tld` → tous les voyants verts.

2. En liste blanche, un serveur **non listé** est refusé (journal Synapse&nbsp;:
   `Federation denied`), un domaine listé fonctionne.

</Steps>

## Inscrire des amis (comptes locaux)

En plus des utilisateurs du SSO, on peut ouvrir une inscription **sur jeton**
pour des comptes locaux (amis). Aucune inscription n'est possible sans jeton.

<Steps>

1. Activer l'inscription sur jeton dans la config nix de l'hôte, puis
   déployer&nbsp;:

   ```nix
   # usr/machines/<host>/default.nix
   darkone.service.matrix.friendRegistration.enable = true;
   ```

2. Promouvoir le compte `matrix.admin` en *server admin* (une seule fois), sur
   l'hôte Matrix&nbsp;:

   ```sh
   sudo -u matrix-synapse psql matrix-synapse \
     -c "UPDATE users SET admin = 1 WHERE name = '@alice:domain.tld';"
   ```

3. Créer un jeton d'inscription via l'API d'administration (avec un *access
   token* du compte admin)&nbsp;:

   ```sh
   curl -X POST \
     -H "Authorization: Bearer <ACCESS_TOKEN>" \
     -H "Content-Type: application/json" \
     -d '{"uses_allowed": 1, "expiry_time": null}' \
     https://matrix.domain.tld/_synapse/admin/v1/registration_tokens/new
   ```

   La réponse contient le `token` à transmettre à l'ami.

4. L'ami s'inscrit dans Element&nbsp;: *Créer un compte* → serveur `domain.tld` →
   il colle le jeton lorsqu'il est demandé.

</Steps>

<Aside type="note" title="L'access token admin">
Récupérer un *access token* dans Element (*Paramètres → Aide & à propos →
Avancé*) avec le compte promu, ou par connexion API
(`POST /_matrix/client/v3/login`). La future interface Synapse-Admin
simplifiera promotion et jetons.
</Aside>

## Salons publics avec validation (knock)

Un salon **public** au sens Matrix se rejoint **librement**&nbsp;: pas de
validation. Pour un salon «&nbsp;joignable mais sur approbation&nbsp;», utiliser
la règle d'accès **knock** (toquer) — réglage **par salon**, côté client, sans
configuration serveur.

<Steps>

1. Dans Element&nbsp;: salon → *Paramètres du salon* → *Sécurité &
   confidentialité*.

2. Accès au salon → **«&nbsp;Demander à rejoindre&nbsp;»** (knock).

3. Un utilisateur (local ou distant fédéré) demande l'accès&nbsp;; un modérateur
   approuve ou refuse.

</Steps>

<Aside type="tip" title="Vraiment privé ?">
Pour un salon confidentiel&nbsp;: le retirer de l'annuaire **et** le passer en
`knock` ou sur invitation. Le chiffrement de bout en bout est déjà actif par
défaut sur les nouveaux salons.
</Aside>

### Diagnostiquer un bot muet

:::note[Vérifications côté client]
Vérifier sur le client (Element, Fractal...) que le bot n'est pas ignoré.
Faire un clique droit sur l'utilisateur correspondant au bot → information
→ sécurité, puis "ignoré". Si le bot réponds bien dans les logs mais que
rien ne se passe, il se peut que le blocage soit côté client.
:::

Un bot qui ne répond pas à `help` alors que son service tourne tombe presque
toujours dans l'un des deux cas&nbsp;: ses tokens ne correspondent plus à la
registration chargée par Synapse, ou il reçoit les messages mais ne peut pas
les **déchiffrer**. Diagnostic dans l'ordre&nbsp;:

<Steps>

1. **Le service tourne-t-il vraiment ?**

   ```sh
   systemctl status mautrix-<pont>
   journalctl -u mautrix-<pont> -n 50
   ```

   S'il redémarre en boucle avec «&nbsp;The as_token was not accepted&nbsp;»,
   l'enregistrement en mémoire de Synapse ne correspond plus aux tokens du
   pont (typique après un reset ou une migration, voir l'encart plus
   haut)&nbsp;: `systemctl restart matrix-synapse`, puis redémarrer le pont.

2. **Le message arrive-t-il au pont ?** Suivre Synapse pendant l'envoi
   d'un `help` au bot&nbsp;:

   ```sh
   journalctl -u matrix-synapse -f | grep transactions
   ```

   Une ligne `PUT .../_matrix/app/v1/transactions/... 200` doit apparaître.
   Si rien ne part, l'appservice n'est pas chargé&nbsp;: vérifier la présence
   de l'enregistrement dans `/var/lib/mautrix-<pont>/` et redémarrer Synapse.

3. **Le pont peut-il le déchiffrer ?** Si la transaction passe (200) mais
   que le bot reste muet, suivre ses logs pendant un nouvel envoi&nbsp;:

   ```sh
   journalctl -u mautrix-<pont> -f
   ```

   Chercher `decrypt`, `session`, `verification`, `dropping`. Un échec de
   déchiffrement signifie que la clé de session du salon n'a pas été
   partagée avec le device actuel du bot (cas classique après un reset du
   pont&nbsp;: son identité e2ee a changé).

4. **Recréer la conversation.** Quitter le message direct avec le bot et en
   démarrer un nouveau&nbsp;: le client établit une session de chiffrement
   neuve et partage la clé avec le device actuel du bot. Vérifier au passage
   que sa propre session Element est **vérifiée**.

5. **Dernier recours**&nbsp;: réinitialiser l'état du pont (encart
   ci-dessous), puis redémarrer Synapse pour recharger l'enregistrement.

</Steps>

<Aside type="note" title="Statuts WhatsApp désactivés">
Le pont WhatsApp ne relaie pas les « statuts » (stories) : l'option
`network.enable_status_broadcast` est forcée à `false`, sans quoi chaque
utilisateur serait réinvité en boucle dans un salon « WhatsApp Status
Broadcast ».
</Aside>

<Aside type="caution" title="Réinitialiser un pont">
Les données d'un pont (sessions, correspondance des salons) vivent dans
`/var/lib/mautrix-*`. Supprimer ce répertoire déconnecte **tous** les
utilisateurs du pont et casse la correspondance des salons existants :
à ne faire qu'en dernier recours, service arrêté.
</Aside>
