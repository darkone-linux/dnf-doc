Chaque fichier de code porte un en-tête court qui explique son rôle. Il aide les
humains comme les agents IA à se repérer sans lire tout le fichier.

{/* t:p 9c3cb4170e266eaa6f0b54757573c357cd543fd947201ed4ee604ab2f89fb8b2 */}
## Règles

- **1re ligne** : description concise du rôle du fichier.
- Décrire **le rôle**, pas l'implémentation ligne à ligne ; expliquer le *pourquoi*.
- Viser l'usage : **utiliser, maintenir, configurer, déboguer**.
- Signaler toute contrainte : fichier généré, à ne pas éditer, dépendances.
- Anglais, concis ; jamais redire ce que le code exprime déjà.

:::caution[Ligne vide avant chaque commentaire]
Une ligne vide **précède toujours** un commentaire (code Nix, heredoc shell,
1er commentaire dans `{` ou `(`). **Seul l'en-tête** en tout début de fichier y
échappe.
:::

## Admonitions dans l'en-tête (Nix)

L'en-tête d'un module Nix peut contenir des admonitions Starlight (`note`,
`tip`, `caution`, `danger`) : elles **alimentent la [référence des
modules](/fr/ref/modules/)** générée par `just codegen`.

```nix
# Module de service : reverse-proxy Caddy pour les services de zone.
#
# :::tip[Activation]
# Activé d'office par les profils d'hôtes exposant des services.
# :::
{ config, lib, ... }:
```

## Fichier généré

Un fichier produit par un outil le **dit en en-tête**, pour qu'on ne l'édite pas :

```nix
# This file is generated by 'just generate' or 'just clean'
# from the configuration file etc/config.yaml
# --> DO NOT EDIT <--
```

:::danger[Ne pas éditer la sortie]
Pour changer un fichier généré, modifier **le générateur ou la source**
(`etc/config.yaml`, le module), puis régénérer. Voir
[Le générateur](/fr/doc/dev-guide/generator/).
:::

:::note[Conventions complètes]
Le style de commentaires et la structure des modules sont décrits dans
`dnf/AGENTS.md` (framework). Côté documentation, voir l'[AGENTS.md](/fr/doc/dev-guide/documentation/) du site.
:::
