Every code file carries a short header explaining its role. It helps humans and
AI agents alike find their way without reading the whole file.

{/* t:p 9c3cb4170e266eaa6f0b54757573c357cd543fd947201ed4ee604ab2f89fb8b2 */}
## Rules

- **1st line**: concise description of the file's role.
- Describe **the role**, not the line-by-line implementation; explain the *why*.
- Target usage: **using, maintaining, configuring, debugging**.
- Flag any constraints: generated file, do not edit, dependencies.
- English, concise; never restate what the code already expresses.

:::caution[Blank line before each comment]
A blank line **always precedes** a comment (Nix code, shell heredoc,
1st comment inside `{` or `(`). **Only the header** at the very top of the file
is exempt.
:::

## Admonitions in the header (Nix)

The header of a Nix module may contain Starlight admonitions (`note`,
`tip`, `caution`, `danger`): they **feed the [module
reference](/en/ref/modules/)** generated by `just codegen`.

```nix
# Service module: Caddy reverse-proxy for zone services.
#
# :::tip[Activation]
# Enabled by default by host profiles exposing services.
# :::
{ config, lib, ... }:
```

## Generated file

A file produced by a tool **states it in the header**, so it is not edited:

```nix
# This file is generated by 'just generate' or 'just clean'
# from the configuration file etc/config.yaml
# --> DO NOT EDIT <--
```

:::danger[Do not edit the output]
To change a generated file, modify **the generator or the source**
(`etc/config.yaml`, the module), then regenerate. See
[The generator](/en/doc/dev-guide/generator/).
:::

:::note[Full conventions]
Comment style and module structure are described in
`dnf/AGENTS.md` (framework). For documentation, see the site's [AGENTS.md](/en/doc/dev-guide/documentation/).
:::
