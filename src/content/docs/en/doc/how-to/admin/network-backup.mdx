The `darkone.service.restic` module backs up fleet hosts with
[restic](https://restic.net/): each host (client) pushes its data to a
repository, either local or remote via a zone REST server.

{/* t:p a7cd15d9ac7449aefb979fb664cf3a60c0ab193427c50b16c9eab26062cdac7e */}
## Principles

Two roles, one module:

- **REST server**: a host that declares the `restic` service in `config.yaml`.
  It stores repositories for all hosts and listens on the internal network.
- **Client**: any host with `enable = true` that declares `targets`
  (backup destinations).

```d2
  style.stroke-dash: 4

```

<Aside type="caution" title="Two passwords not to confuse">
- **Repository passphrase** `restic-password-<zone>`: encrypts the *content* of the
  repository. Without it, no restoration is possible.
- **REST credential** `restic/<host>/rest-password`: *authenticates* the host
  to the REST server. One per host.

Both are generated by `just passwd-restic` (see below).
</Aside>

Repository layout, by category:

```
<root>/<host>/system      # "system" category (/ minus exclusions)
<root>/<host>/srv/nfs     # "nfs" category    (/srv/nfs/<...>)
<root>/<host>/srv/medias  # "media" category  (/srv/medias/<...>)
```

## Setting up a backup server

<Steps>

1. In `etc/config.yaml`, add the `restic` service to the server host:

   ```yaml
   services:
     restic:
   ```

2. Attach the backup disk and mount it on `/mnt/backup` in the host
   configuration (`usr/machines/<host>/`), with the `nofail` option:

   ```nix
   fileSystems."/mnt/backup" = {
     device = "/dev/disk/by-uuid/<uuid>";
     fsType = "ext4";
     options = [ "nofail" ];
   };

   # The REST server only starts once the disk is mounted.
   systemd.services.restic-rest-server = {
     requires = [ "mnt-backup.mount" ];
     after = [ "mnt-backup.mount" ];
   };
   ```

3. The repository is stored under `serverDataDir` (default `/mnt/backup/restic`).
   The firewall opens the REST port on the internal interface automatically.

</Steps>

<Aside type="note" title="Server is also a client">
A server can back up its own data by pointing a `target` to its local disk
(`root = "/mnt/backup/restic"`).
</Aside>

## Adding a client

Declare one or more `targets` in the host configuration:

```nix
darkone.service.restic = {
  enable = true;
  targets = [
    {
      # Remote repository via the zone REST server.
      root = "rest:http://restic.zone.domain.tld:8888";
      zone = "ag";
      categories = [ "system" "nfs" ];
    }
  ];
};
```

`target` fields:

| Field        | Role                                                        |
| ------------ | ----------------------------------------------------------- |
| `name`       | Identifier (default `main`), suffix for jobs/units.         |
| `root`       | Local path or URL `rest://`.                                |
| `zone`       | Zone selecting the `restic-password-<zone>` passphrase.     |
| `categories` | Data to back up: `system`, `nfs`, `medias`.                 |

<Aside type="tip" title="Multiple destinations">
Add a second `target` (different `name`, different `root`/`zone`) to replicate
the backup, for example to the REST server of another zone.
</Aside>

## Generating passwords

An idempotent recipe creates missing secrets (never overwrites):

```bash
just passwd-restic
```

It generates, in `usr/secrets/secrets.yaml`:

- one `restic-password-<zone>` passphrase per zone;
- one `restic/<host>/rest-password` credential per fleet host.

<Aside type="note" title="New host">
Re-run `just passwd-restic` after adding a host, then redeploy the
servers so they pick up its new REST account.
</Aside>

## Hardening repository isolation

By default, the server allows any authenticated host to access any
repository. The `enableServerPrivateRepos` option enforces that each host can
only access its own repository (prefix = hostname). Activate it **in two steps**
to avoid any disruption:

<Steps>

1. Deploy **the entire fleet** with per-host credentials (clients + servers),
   `enableServerPrivateRepos = false`. Verify that backups work.
2. Set `enableServerPrivateRepos = true` and redeploy **the servers**.

</Steps>

## Restoring

On the target host, secrets are already in place. List then restore:

```bash
# REST access variables (generated by sops) and repository passphrase.
set -a; . /run/secrets/rendered/restic-rest-env; set +a
export RESTIC_PASSWORD_FILE=/run/secrets/restic-password-ag
export RESTIC_REPOSITORY=rest:http://restic.zone.domain.tld:8888/$(hostname)/system

restic snapshots
restic restore latest --target /restore
```

<Aside type="caution" title="Adjust zone and category">
Replace `zone` with the repository zone and `system` with the category
(`srv/nfs`, `srv/medias`) to restore.
</Aside>

## Verifying

<Steps>

1. On a server, one account per fleet host in the htpasswd file:

   ```bash
   sudo cut -d: -f1 /run/restic-rest/htpasswd
   ```

2. On a client, run an on-demand backup:

   ```bash
   systemctl start restic-backups-system-main.service
   journalctl -u restic-backups-system-main.service -e
   ```

3. Isolation (if `enableServerPrivateRepos = true`): access to another host's
   repository must be **denied** (401/403 error).

</Steps>
