Backups rely on [restic](https://restic.net/): each host **pushes** its data to an encrypted repository, either local or remote via a zone REST server. The goal is a **3-2-1** strategy (multiple copies, multiple media, one off-site location).

{/* t:p 60dc1832f21e2477c514f34bd29ffb2ab547b0006887f74cb5e17c4c56bf9f1b */}
## Two roles, one module

```d2
style.fill: transparent
direction: down

c1: Client host\nrestic backup { shape: person }
c2: Client host\nrestic backup { shape: person }
srv: REST server\nzone repositories { shape: cylinder }

c1 -> srv
c2 -> srv
```

| Role | Who | What |
|---|---|---|
| REST server | Host with the `restic` service | Stores zone repositories |
| Client | Host with `enable = true` | Pushes its `targets` (backups) |

{/* t:p f641464c0b5a23eef37f17c493c463e28feb2e0f8fef3c81b894c0d0b1ff6acb */}
## Backup categories

A target backs up one or more **categories** into sub-repositories:

| Category | Content |
|---|---|
| `system` | Root `/`, excluding volatile data and `/nix`, `/srv`… |
| `nfs` | Shares `/srv/nfs/...` |
| `medias` | Media `/srv/medias/...` |

:::note[Two secrets, not to be confused]
- **Repository passphrase** (`restic-password-<zone>`) — encrypts the content.
- **REST credential** (`restic/<host>/rest-password`) — authenticates the host.

Both are generated by `just passwd-restic`. See
[Secrets](/en/doc/admin-guide/operate/secrets/).
:::

## Implementation

The full operational configuration (REST server, targets, retention policy, restoration) is detailed in the how-to guide:

- [Network backups (restic)](/en/doc/how-to/admin/network-backup/)

:::tip[Retention policy]
Default: 24 last + 24 hourly + 7 daily + 8 weekly + 24 monthly. Integrity is verified on each backup.
:::
