The DNF network operates at two scales: an isolated local **zone**, then
multiple zones connected by a mesh **VPN**. The vocabulary is defined in
[Concepts](/en/doc/admin-guide/understand/concepts/).

{/* t:p dc9534f835e0e09db41110a71aa8e46e59668b5a9a710b8379ec1c2ae465ed93 */}
## Simple network (one zone)

A zone is a gateway and its machines.

```d2
style.fill: transparent
direction: down

net: Internet { shape: cloud }

gw: Gateway {
  fw: NAT + Firewall
  dns: dnsmasq\nDNS + DHCP
  fw -> dns
}

lan: Local zone {
  pc: Workstations
  srv: Server { shape: cylinder }
}

net -> gw.fw: WAN
gw.dns -> lan.pc: LAN
gw.dns -> lan.srv: LAN
```

- The gateway shares Internet (NAT) and protects the zone (firewall).
- `dnsmasq` distributes addresses (DHCP) and resolves local names
  (`<host>.<zone>`).
- On the Internet side (WAN), everything is closed. On the local side (LAN), only
  SSH, DNS, DHCP and the home page are open.

## Ad blocking (AdGuardHome)

Optional service, enabled on the gateway. It inserts itself in front of `dnsmasq`.

```d2
style.fill: transparent
direction: down

pc: Workstation

agh: AdGuardHome\nport 53, filtering { shape: hexagon }

dns: dnsmasq\nlocal names { shape: cylinder }
up: External DNS\nencrypted { shape: cloud }

pc -> agh: DNS query
agh -> dns: Local names
agh -> up: Rest, filtered
```

- AdGuardHome becomes the DNS resolver for workstations (port 53).
- It blocks ads and trackers, and encrypts queries to the outside.
- `dnsmasq` moves to an internal port and keeps local names and DHCP.

## Extended network (VPN)

Multiple zones are connected by a mesh VPN, coordinated by the coordination server
(HCS).

```d2
style.fill: transparent
direction: down

hcs: Coordination\nHeadscale (HCS) { shape: cylinder }

zoneA: Zone A {
  gwa: Gateway
  la: Workstations
  gwa -> la
}
zoneB: Zone B {
  gwb: Gateway
  lb: Workstations
  gwb -> lb
}
ext: Remote machine

hcs -> zoneA.gwa: VPN
hcs -> zoneB.gwb: VPN
hcs -> ext: VPN
zoneA.gwa <-> zoneB.gwb: Mesh
```

- The HCS (Headscale) is public and orchestrates the mesh.
- Each gateway is a *subnet router*: it publishes its zone's subnet into the
  VPN.
- Zones can see each other; remote machines (smartphone, PC) join the network
  via Tailscale.
- A forwarder resolver on the HCS routes each zone to its own DNS.

<Aside type="note" title="Optional VPN">
The mesh is only enabled if `network.coordination.enable` is true in
`etc/config.yaml`. Without it, zones remain local and independent.
</Aside>

## DNS names and resolution

Three families of names coexist on the network. All are generated from
`etc/config.yaml`.

| Name | Form | Scope | Example |
|---|---|---|---|
| **Simple name** | `<host>` | Entire internal network | `server` |
| **Zone FQDN** | `<name>.<zone>.<domain>` | Entire internal network | `cloud.home.domain.tld` |
| **Global service** | `<name>.<domain>` | Internet included | `git.domain.tld` |

- **Simple names** : these are the **machines** (declared hosts). Resolved
  everywhere on the network, to the machine's real address, regardless of its
  zone.
- **Zone FQDNs** : these are the **services of a zone**. They point to the
  gateway of that zone, which serves them via its reverse proxy.
- **Global services** : services marked `global` in `etc/config.yaml`.
  Exposed on the HCS with **public DNS** : reachable from anywhere.

Who resolves what :

- `dnsmasq`, on **each gateway**, knows all internal names (machines
  and services of **all** zones) : it resolves simple names and FQDNs.
- The current zone is resolved locally ; other zones and the global domain
  are routed to the correct gateway, or to the HCS pivot resolver.
- Public (Internet) names go to upstream encrypted DNS, filtered by
  AdGuardHome if enabled.

{/* t:p c9144db0481f8a97ed7d130a099ca5cf98925d1e1e53b8cba1f734e4a2513e0a */}
### The pivot resolver (HCS)

On the coordination server (HCS), a **pivot resolver** centralizes the internal DNS resolution of the VPN. It knows the zone layout and merely routes&nbsp;:

- each zone FQDN to its zone's gateway&nbsp;;
- the rest to an encrypted external DNS.

All VPN machines query it on the HCS's VPN address. It bridges zones together for name resolution.

### Resolution from outside

A machine outside any zone (laptop on the go, smartphone) joins the network via
the VPN. Its DNS resolution is **split** (split-DNS) according to the type of name
requested:

- **internal names**: routed, through the VPN, to the HCS hub resolver, which
  then directs them to the correct gateway;
- **public names**: resolved by the local resolver of the visited network (ISP
  box, public access point), without going through the VPN.

```d2
style.fill: transparent
direction: down

pc: Roaming device { shape: person }
split: Split-DNS routing { shape: diamond }

pivot: Hub resolver\nHCS { shape: cylinder }
gwa: Zone A gateway
gwb: Zone B gateway
local: Local resolver\nof the visited network { shape: cloud }
net: Internet { shape: cloud }

pc -> split
split -> pivot: Internal names\n(via VPN)
split -> local: Public names
pivot -> gwa: Zone A
pivot -> gwb: Zone B
local -> net
```

:::note[Why not route public names through the VPN]
Routing everything through the HCS resolver tunnels all DNS traffic inside the
VPN. If the VPN path degrades (intermediate relay, blocked UDP), nothing resolves
anymore. Leaving public names to the local resolver preserves Internet access even
when the VPN link is unstable.
:::

:::tip[Device reconnected in a zone]
A roaming laptop reconnected via ethernet in a zone no longer needs the VPN to
reach it. Its VPN client then pauses to hand control back to the local gateway
(direct resolution and routing), then reactivates upon departure.
:::
