For day-to-day operation, see

## Overview

Everything is handled by `dnf/modules/service/matrix.nix`, with two satellite
modules&nbsp;: `element.nix` (static web client) and `turn.nix` (coturn relay
for audio/video).

```d2
style.fill: transparent
direction: down

client: Clients\n(Element, Fractal) {
  shape: person
}
caddy: Caddy\n(reverse proxy)
synapse: Synapse {
  shape: rectangle
}
pg: PostgreSQL {
  shape: cylinder
}
idm: Kanidm\n(OIDC)
turn: coturn\n(TURN/STUN)

bridges: mautrix bridges {
  wa: whatsapp
  sig: signal
  tg: telegram
  meta: messenger
  dc: discord
}

ext: External networks\n(WhatsApp, Telegram...) {
  shape: cloud
}

client -> caddy -> synapse -> pg
synapse -> idm
client -> turn
synapse <-> bridges <-> ext
```

- **Caddy** routes `/_matrix/*` and `/_synapse/client/*` to Synapse and serves
  `/.well-known/matrix/{client,server}` files (client discovery and federation).
- **Synapse** listens on `dnfConfig.network.ports.matrix`, local PostgreSQL
  database (created with `C` collation by a `matrix-db-init` unit).
- **Kanidm** provides authentication&nbsp;: OIDC client provisioned via
  `darkone.service.idm.oauth2.matrix`, *local part* derived from
  `preferred_username` (see [Authentication & IDM](/en/doc/admin-guide/understand/auth-idm/)).
- **coturn** is wired automatically if the `turn` service is declared
  (`turn_uris`, shared secret via sops).

## Module anatomy

`matrix.nix` is a `lib.mkMerge` of independent blocks:

| Block | Condition | Content |
|---|---|---|
| Base | always | OIDC client, reverse proxy, persistence |
| Server | `matrix.enable` | Synapse, PostgreSQL, common secrets, `doublepuppet` appservice |
| One block per bridge | `matrix.enable && bridges.<x>.enable` | sops secrets + bridge's mautrix service |

Each bridge can thus be disabled individually
(`darkone.service.matrix.bridges.{whatsapp,signal,telegram,messenger,discord}.enable`),
its sops secrets only being declared if the bridge is active.

## Mautrix bridges

DNF relies on the nixpkgs `services.mautrix-*` modules, which generate the
appservice *registration* and register it with Synapse on startup.
Two generations of bridges coexist, with different configuration keys:

| Bridge | Generation | Double puppeting | Level |
|---|---|---|---|
| whatsapp | bridgev2 (Go) | `double_puppet.secrets` | `user` |
| signal | bridgev2 (Go) | `double_puppet.secrets` | `user` |
| messenger (meta) | bridgev2 (Go) | `double_puppet.secrets` | `user` |
| telegram | legacy (Python) | `bridge.login_shared_secret_map` | `full` |
| discord | legacy (Go) | `bridge.login_shared_secret_map` | `user` |

<Aside type="note" title="Why « full » for Telegram?">
On the legacy Telegram bridge, the `user` level doesn't allow connecting one's
own account (at least `puppeting` is required). DNF gives `full` to the domain
so that every local account can connect and share their rooms.
</Aside>

### Permissions

The `mkBridgePermissions` helper (in `matrix.nix`) builds the same
policy for all bridges:

```yaml
permissions:
  "domain.tld": user          # tout compte local, avec son propre compte distant
  "@alice:domain.tld": admin  # le matrix.admin déclaré dans etc/config.yaml
```

Distant sessions are isolated per Matrix user: generalizing to all
domain accounts carries no risk of interference.

### Double puppeting

Official [appservice](https://docs.mau.fi/bridges/general/double-puppeting.html)&nbsp;:
a `doublepuppet` appservice is registered in Synapse
(`app_service_config_files`) with a namespace covering `@.*:domain.tld`.
Its `as_token` allows each bridge to emit events **on behalf of the user's
real account**.

```d2
style.fill: transparent
direction: down

phone: Phone\n(WhatsApp) {
  shape: person
}
bridge: mautrix bridge
synapse: Synapse
room: Matrix room\n(@alice's message) {
  shape: page
}

phone -> bridge: outgoing message
bridge -> synapse: as_token doublepuppet
synapse -> room
```

The token is unique for all bridges (`mautrix-doublepuppet-as-token`) and
injected into each config by environment variable substitution
(`as_token:$MAUTRIX_DOUBLEPUPPET_AS_TOKEN`), with nixpkgs modules passing the
configuration through `envsubst` via the service's `environmentFile`.

## Secrets and ports

- **Secrets** : declared per block in `matrix.nix` (`sops.secrets.*`), and
  assembled into environment files by `sops.templates.*` owned
  by the bridge's system user. Full list in
  [la page d'exploitation](/en/doc/admin-guide/operate/matrix/#required-secrets).
- **Ports** : `matrix` and `matrixTelegram` are set by the `dnf/config/network.nix`
  registry; the default appservice ports of the other bridges
  (29318, 29319, 29328, 29334) are listed there as `reserved` to avoid
  any future collision.
- **WhatsApp Statuses** : `network.enable_status_broadcast = false` in the
  bridge config, otherwise the bridge indefinitely recreates
  the "WhatsApp Status Broadcast" room for each user.
