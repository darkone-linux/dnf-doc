The fleet is deployed with [colmena](https://github.com/zhaofengli/colmena): a single command builds and applies the configuration on one or more hosts, remotely, from the admin workstation.

{/* t:p d4a06fc139c0d5029ddc5205311c422d5da5a5e1fe1b2e6d014d42aae5b8629c */}
## Apply a configuration

```sh
just apply <target> [action]    # alias : a
```

- **target** = hostname, pattern (`'*'`), list (`a,b`) or colmena tag (`@server`).
- **action** = `switch` (default), `boot`, `test` or `build`.

| Command | Purpose |
|---|---|
| `just apply <target>` | Build + activate on the target(s) |
| `just apply-local` | Apply on the current machine (alias `al`) |
| `just apply-verbose <target>` | Same as `apply` in trace mode (alias `av`) |

:::tip[Build on target]
`apply` builds **on the target host** (`--build-on-target`): the admin workstation doesn't need the power or architecture of the deployed machines.
:::

## Deploy safely

**Build confidence through action**: each step is less risky than the next.

```sh
just apply <host> build      # downloads + compiles: 100% safe, nothing is activated
just apply <host> test       # activates without switching: no generation, no boot
just apply <host>            # switch: activates and creates a new generation
```

**Progress from core to edge**: deploy nodes in this order, to never cut off
access to a node by the one preceding it.

```d2
style.fill: transparent
direction: right

hcs: HCS
gwl: Local gateway
hl: Local hosts
gwd: Remote gateways
hd: Remote hosts

hcs -> gwl -> hl -> gwd -> hd
```

:::caution[Why this order]
A remote host's connectivity depends on its gateway, which depends on the VPN
coordinated by the HCS. Deploying a link before its support can make subsequent
hosts unreachable.
:::

## Update

The system is pinned by flakes. Updating = refresh the inputs, then redeploy.

```sh
just update-flake     # updates dnf/ + root, commits the locks
just apply '*'        # deploys the update
```

:::caution[Always test first]
Before a large deployment, validate locally: `just check-flake` (or
`just check-all`), then `just apply <pilot-host>` on **one** test host.
:::

## Binary cache

Two services work together to avoid recompiling or re-downloading
derivations at every deployment.

| Service | Role |
|---|---|
| `harmonia` | Serves the local, signed `/nix/store` **directly** to hosts (LAN, or VPN if `global`). |
| `nix-cache` | Per-zone `nginx` proxy that caches the public `cache.nixos.org` cache on the gateway. |

A host therefore fetches a derivation in this order: its zone's `harmonia`, any
`global` `harmonia`, then its zone's `nix-cache` proxy.
The first fetch from the Internet is thus shared for the entire zone.

:::tip[Roaming host (multi-zone laptop)]
A laptop that changes zones must not keep fixed substituters from its declared
zone: unreachable elsewhere, they cause timeouts on every request. Enable
*roaming* mode:

```nix
# usr/machines/<hĂ´te>/default.nix
darkone.service.nix-cache.roaming = true;
```

Its substituters become **neutral** names (`harmonia.dnf.internal`,
`nix-cache.dnf.internal`) that each zone's DNS resolves to its own caches â†’ the
cache follows the machine. Outside any zone, immediate NXDOMAIN response â†’
direct fallback on `cache.nixos.org`, no delay. Since signing keys are shared
across the entire network, nothing else to configure.
:::

:::note[Activation]
`harmonia` and `nix-cache` are [services](/en/doc/admin-guide/operate/services/)
to enable on a network host (`nix-cache` on the zone gateway). harmonia's
signing key is managed by
[sops](/en/doc/admin-guide/operate/secrets/) (`just configure-admin-host`).
:::

:::tip[Why a single-upstream proxy]
`nix-cache` only relays **one** upstream (`cache.nixos.org`): it caches
`narinfo` and `nar` as-is, signatures intact. No URL rewriting or database, so
no HTTP 500 "invalid nar hash" error from the old `ncps` relay.
:::
