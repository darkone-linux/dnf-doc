Enable the Matrix server (Synapse), its Element web client and mautrix
bridges (WhatsApp, Signal, Telegram, Messenger, Discord). For the architecture, see
[Matrix in DNF](/en/doc/admin-guide/understand/matrix/).

{/* t:p 8026b991054a535675002db4a5475a7412eb9eaaeec602690f5d2a5f48c5ae27 */}
## Enable Matrix

<Steps>

1. Declare the services on the target host in `etc/config.yaml`&nbsp;:

   ```yaml
   # etc/config.yaml
       services:
         matrix:
           title: "Matrix"
           description: "Messagerie instantanée"
         element:        # client web
         turn:           # audio/video relay (recommended)
   ```

2. Declare the Matrix administrator (global key, outside hosts)&nbsp;:

   ```yaml
   # etc/config.yaml
   matrix:
     admin: "alice"    # local part of the admin account
   ```

   This account administers the bridges and receives
   [supervision alerts](/en/doc/admin-guide/operate/alerts/).

3. Create the required [secrets](/en/doc/admin-guide/operate/secrets/)
   (see table below), then deploy.

</Steps>

Authentication goes through [SSO Kanidm](/en/doc/admin-guide/operate/sso/)
(OIDC)&nbsp;: every declared user connects with their network account,
no Matrix account creation is needed.

{/* t:p 612e4b4b4c6fdc9f568eccb61bb4176b608d5b885eb2f88cbbb765f0d7e6f941 */}
### Required secrets

To fill in `usr/secrets/secrets.yaml`&nbsp;:

| Sops key | Role | Generation |
|---|---|---|
| `matrix-rss-password` | Synapse registration shared secret | `openssl rand -hex 32` |
| `oidc-secret-matrix` | OIDC Kanidm client secret | see [SSO](/en/doc/admin-guide/operate/sso/) |
| `turn-secret` | coturn shared secret (if `turn` enabled) | `openssl rand -hex 32` |
| `mautrix-doublepuppet-as-token` | double puppeting appservice token (shared by bridges) | `openssl rand -hex 32` |
| `mautrix-doublepuppet-hs-token` | Same, required by Synapse, never used | `openssl rand -hex 32` |

## Managing bridges

WhatsApp, Signal, Telegram and Messenger bridges are **enabled by
default** with Matrix&nbsp;; Discord is **disabled by default**. Per-host setting in
the machine's nix configuration&nbsp;:

```nix
# usr/machines/<host>/default.nix
darkone.service.matrix.bridges = {
  discord.enable = true;     # opt-in
  messenger.enable = false;  # disable a default bridge
};
```

Each bridge has its own secrets&nbsp;:

| Bridge | Sops keys | Note |
|---|---|---|
| WhatsApp | `mautrix-whatsapp-as-token`, `mautrix-whatsapp-hs-token`, `mautrix-whatsapp-encryption-pickle-key` | `openssl rand -hex 32` each |
| Signal | `mautrix-signal-as-token`, `mautrix-signal-hs-token`, `mautrix-signal-encryption-pickle-key` | `openssl rand -hex 32` each |
| Messenger | `mautrix-meta-as-token`, `mautrix-meta-hs-token`, `mautrix-meta-encryption-pickle-key` | `openssl rand -hex 32` each |
| Telegram | `mautrix-telegram-api-id`, `mautrix-telegram-api-hash`, `mautrix-telegram-as-token`, `mautrix-telegram-hs-token` | API id/hash to create on [my.telegram.org](https://my.telegram.org) |
| Discord | `mautrix-discord-as-token`, `mautrix-discord-hs-token` | `openssl rand -hex 32` each |

The `as`/`hs` tokens make each bridge's appservice *registration*
deterministic (see
[appservice tokens](/en/doc/admin-guide/understand/matrix/#appservice-tokens)).

<Aside type="caution" title="Migrating an already deployed bridge">
On a WhatsApp or Signal bridge deployed **before** these tokens were
introduced, the existing registration keeps the old auto-generated tokens. To switch
to sops tokens&nbsp;: delete `/var/lib/mautrix-<pont>/<pont>-registration.yaml`,
restart the bridge (the registration is regenerated from the config), then
restart `matrix-synapse`. For Messenger and Discord, a simple redeployment
followed by a Synapse restart is enough&nbsp;: their modules rewrite the
registration from the config at every start.
</Aside>

{/* t:p f3eb4bb567f7ef6cafb1fd0f14a2f32d19ccfd22540309eef5aaad8f5d9ac836 */}
### Permissions

Nothing to manage&nbsp;: any local account (`@*:domain.tld`) can use each
bridge with **their own** remote account, without interference between
users. The `matrix.admin` account has the bot administration
commands (`help` in direct conversation with the bot for the
list).

{/* t:p 57866a0b0c836e99e3a85df4799b3a07fa81c8c06bcd8807ab57b6fae2e5fbb9 */}
### Double puppeting

Bridges share a `doublepuppet` appservice (token
`mautrix-doublepuppet-as-token`)&nbsp;: messages sent from the
phone appear in Matrix as coming from the user's **real account**.
This is automatic for all local accounts, no action per
user.

## Opening the network to federation

By default, the server federates with **all** Matrix servers, while remaining
**invisible in searches** (directories not exposed). To control the scope, set
`darkone.service.matrix.federation` in the host's nix config:

```nix
# usr/machines/<host>/default.nix
darkone.service.matrix.federation = {
  enable = true;       # false = federation fully blocked
  whitelist = [ ];     # empty = open ; otherwise whitelist
};
```

| Goal | Setting |
|---|---|
| Isolated network (no federation) | `enable = false;` |
| Open to the entire Matrix ecosystem | `enable = true; whitelist = [ ];` |
| Restricted to trusted servers | `enable = true; whitelist = [ "ami.org" "matrix.org" ];` |

<Aside type="caution" title="Family network">
The whitelist is **bidirectional**: an unlisted server can neither write to
your users nor receive from them. For a network used by children, fill in
`whitelist` rather than leaving federation open.
</Aside>

Verify after deployment:

<Steps>

1. Test incoming federation with
   [federationtester.matrix.org](https://federationtester.matrix.org/) on
   `domain.tld` → all green lights.

2. With a whitelist, an **unlisted** server is denied (Synapse log:
   `Federation denied`), a listed domain works.

</Steps>

## Registering friends (local accounts)

In addition to SSO users, you can enable **token-based** registration for local
accounts (friends). No registration is possible without a token.

<Steps>

1. Enable token-based registration in the host's nix config, then deploy:

   ```nix
   # usr/machines/<host>/default.nix
   darkone.service.matrix.friendRegistration.enable = true;
   ```

2. Promote the `matrix.admin` account to *server admin* (one time only), on
   the Matrix host:

   ```sh
   sudo -u matrix-synapse psql matrix-synapse \
     -c "UPDATE users SET admin = 1 WHERE name = '@alice:domain.tld';"
   ```

3. Create a registration token via the admin API (with an *access token* from
   the admin account):

   ```sh
   curl -X POST \
     -H "Authorization: Bearer <ACCESS_TOKEN>" \
     -H "Content-Type: application/json" \
     -d '{"uses_allowed": 1, "expiry_time": null}' \
     https://matrix.domain.tld/_synapse/admin/v1/registration_tokens/new
   ```

   The response contains the `token` to pass on to the friend.

4. The friend registers in Element: *Create account* → server `domain.tld` →
   they paste the token when prompted.

</Steps>

<Aside type="note" title="The admin access token">
Retrieve an *access token* in Element (*Settings → Help & About → Advanced*)
with the promoted account, or via API login
(`POST /_matrix/client/v3/login`). The future Synapse-Admin interface will
simplify promotion and tokens.
</Aside>

## Next-generation authentication (MAS)

The [Matrix Authentication Service](https://element-hq.github.io/matrix-authentication-service/)
replaces Synapse's internal authentication (required by Element X). For what
it changes and how it fits in, see
[architecture](/en/doc/admin-guide/understand/matrix/#next-generation-authentication-mas).

:::danger[Under validation]
Implemented but not yet proven in production: migration of an
existing server (`syn2mas`) and the `mas-cli` tooling on the host remain to be
validated. In the meantime, leave `mas.enable = false` (default):
classic OIDC authentication remains fully functional.
:::

<Steps>

1. Create the dedicated secrets in `usr/secrets/secrets.yaml`:

   | SOPS key | Role | Generation |
   |---|---|---|
   | `mas-encryption-secret` | MAS internal encryption | `openssl rand -hex 32` |
   | `mas-synapse-secret` | Shared secret MAS ↔ Synapse | `openssl rand -hex 32` |
   | `mas-rsa-private-key` | Token signing key | `openssl genpkey -algorithm RSA -pkeyopt rsa_keygen_bits:4096` |

2. Enable the option in the host nix config, then deploy:

   ```nix
   # usr/machines/<host>/default.nix
   darkone.service.matrix.mas.enable = true;
   ```

</Steps>

:::danger[Immutable secret]
`mas-encryption-secret` must **never** change after the first start
of MAS: it protects encrypted data in the database, which would become
unreadable.
:::

### Migrating an existing production server

Switching an existing Synapse requires a **maintenance window**&nbsp;: MAS
becomes the owner of accounts and sessions, which must be imported with the
official [syn2mas](https://element-hq.github.io/matrix-authentication-service/setup/migration.html)
tool (accounts, passwords, sessions, OIDC links). Without this migration, all
users are logged out and buddy accounts are lost. Key points&nbsp;:

- back up PostgreSQL first (migration is not reversible)&nbsp;;
- `syn2mas check` then `--dry-run` run **without stopping** Synapse&nbsp;;
- remove bridge registrations (`/var/lib/mautrix-*/*-registration.yaml`)
  so they regenerate with the MSC4190 flag, then restart Synapse.

{/* t:p 3a8e25f5d597286618ecde21c708f10c789647a68a72ad8662f704ad5ff0119d */}
### Registering buddies with MAS

Registration tokens are no longer handled through Synapse's admin API&nbsp;:
they are created on the Matrix host with `mas-cli`
(`manage issue-user-registration-token`), and promoting an admin account
becomes `mas-cli manage promote-admin`. The detailed procedure will be
documented once the tooling is validated (see the box above).

## Public rooms with approval (knock)

A **public** room in the Matrix sense can be joined **freely**: no validation.
For a room that is "joinable but on approval", use the **knock** access rule —
a **per-room** setting, on the client side, no server configuration needed.

<Steps>

1. In Element: room → *Room Settings* → *Security & Privacy*.

2. Room access → **"Ask to join"** (knock).

3. A user (local or federated remote) requests access; a moderator approves or
   denies.

</Steps>

<Aside type="tip" title="Really private?">
For a confidential room: remove it from the directory **and** set it to `knock`
or invitation only. End-to-end encryption is already active by default on new
rooms.
</Aside>

### Diagnosing a silent bot

:::note[Client-side checks]
Check on the client (Element, Fractal...) that the bot is not ignored.
Right-click on the bot's user → information
→ security, then "ignored". If the bot replies in the logs but
nothing happens, the block may be on the client side.
:::

A bot that does not reply to `help` while its service is running almost
always falls into one of two cases&nbsp;: its tokens no longer match the
registration loaded by Synapse, or it receives messages but cannot
**decrypt** them. Diagnose in order&nbsp;:

<Steps>

1. **Is the service actually running ?**

   ```sh
   systemctl status mautrix-<bridge>
   journalctl -u mautrix-<bridge> -n 50
   ```

   If it keeps restarting with «&nbsp;The as_token was not accepted&nbsp;»,
   Synapse's in-memory registration no longer matches the bridge's tokens
   (typical after a reset or migration, see the box above)&nbsp;:
   `systemctl restart matrix-synapse`, then restart the bridge.

2. **Does the message reach the bridge ?** Follow Synapse while sending
   a `help` to the bot&nbsp;:

   ```sh
   journalctl -u matrix-synapse -f | grep transactions
   ```

   A `PUT .../_matrix/app/v1/transactions/... 200` line should appear.
   If nothing goes out, the appservice is not loaded&nbsp;: check for the
   registration file in `/var/lib/mautrix-<bridge>/` and restart Synapse.

3. **Can the bridge decrypt it ?** If the transaction goes through (200)
   but the bot stays silent, follow its logs while sending a new message&nbsp;:

   ```sh
   journalctl -u mautrix-<bridge> -f
   ```

   Look for `decrypt`, `session`, `verification`, `dropping`. A
   decryption failure means the room's session key has not been
   shared with the bot's current device (typical after a bridge
   reset&nbsp;: its e2ee identity has changed).

4. **Recreate the conversation.** Leave the direct message with the bot and
   start a new one&nbsp;: the client establishes a fresh encryption session
   and shares the key with the bot's current device. Also check
   that your own Element session is **verified**.

5. **Last resort**&nbsp;: reset the bridge state (box
   below), then restart Synapse to reload the registration.

</Steps>

<Aside type="note" title="WhatsApp statuses disabled">
The WhatsApp bridge does not relay « statuses » (stories): the
`network.enable_status_broadcast` option is forced to `false`, otherwise
every user would be repeatedly re-invited to a « WhatsApp Status
Broadcast » room.
</Aside>

<Aside type="caution" title="Resetting a bridge">
A bridge's data (sessions, room mappings) lives in
`/var/lib/mautrix-*`. Deleting this directory disconnects **all**
bridge users and breaks existing room mappings:
only do this as a last resort, with the service stopped.
</Aside>
