This page summarizes **which ANSSI rules are activated**, according to the **level** and
**category** of a host. It is the reference to consult before hardening a machine.

:::tip[Understand first]
The concepts used here (levels, categories, labels, MAC, egress‚Ä¶) are
explained in [Security
notions](/en/doc/admin-guide/understand/security-notions/).
:::

## Reminder: level and category

```nix
darkone.system.security = {
  enable   = true;
  level    = "intermediary";   # minimal | intermediary | reinforced | high
  category = "server";         # base | client | server
};
```

Levels are **cumulative**: `intermediary` adds its rules to those of
`minimal`, and so on. The category adds the rules specific to a workstation
(`client`) or a server (`server`).

{/* t:p 59d6590db674be9de1f7fac85f5ce3c28eac555ac91ff1eddf3e6ad3f14a02b5 */}
## Recommended category per profile

| Profile | Recommended category | Why |
|---|---|---|
| `minimal`, `portable` | `base` | Universal base |
| `desktop`, `laptop` | `client` | Screen, USB, interactive session |
| `server`, `gateway`, `hcs`, `vm` | `server` | Exposed services, network |

:::caution[Today: `base` everywhere]
Automatic profile ‚Üí category mapping is not yet active. All hosts are
set to `minimal` / `base` by default. To apply a category, set
`category` manually on the relevant host.
:::

## Adapt without breaking everything

Two safety valves allow bypassing a specific protection without lowering the
global level.

- **Tags** (`excludes`): exclude a **group** of incompatible rules
  (e.g. `needs-jit`, `no-ipv6`).
- **Exceptions** (`exceptions`): bypass **one** rule, justification
  required.

Details on these safety valves are in [Security
concepts](/en/doc/admin-guide/understand/security-notions/#exclusion-labels).

:::danger[Module under stabilization]
Raise the level **progressively** and test each host before production.
:::
