Every network member can associate a **YubiKey** primary key and a **backup**
key with their account, in addition to their password (which remains mandatory
and serves as automatic fallback).

The key principle: **one single enrollment** per key, on the administration
machine. Everything else propagates declaratively with `just apply` — the key
is never plugged into the fleet hosts, only when it needs to be used.

{/* t:p 8464395d7306c39267ac2411ba3e63f6cda37aa32b168b8390b7db2cbb1a18d9 */}
## What a key covers

| Usage | Mechanism | Effect |
|---|---|---|
| **Session, sudo, login screen** | pam_u2f (`sufficient`) | Touching the key replaces the password |
| **Encrypted disks (LUKS)** | FIDO2 `hmac-secret` | Unlock at boot by touching the key |
| **SSO (Kanidm)** | Passkey WebAuthn | Passwordless login to services |
| **Vaultwarden** | WebAuthn (2FA) | Second factor for the vault |

:::note[No lockout risk]
The session password (sops) and the install LUKS passphrase are **never
removed**: without a key (lost, forgotten), everything works as before.
:::

## Enroll a key

<Steps>

1. Plug the user's YubiKey into the admin workstation, then:

   ```sh
   just yubikey <login>          # main key (FIDO2 PIN + ~3 touches)
   just yubikey <login> backup   # backup key (optional)
   ```

   If the key does not have a FIDO2 PIN yet, the recipe creates it first
   (required for Kanidm, see below).

2. Commit and deploy:

   ```sh
   git add usr/secrets/yubikeys.json
   just commit "user(yubikey): <login>"
   just apply
   ```

3. That's it: session and sudo respond to the key on all of the user's
   workstations, and encrypted hosts enroll the key into their LUKS headers
   **themselves**, at deployment time.

</Steps>

:::tip[Kanidm and Vaultwarden: a web ceremony, once only]
WebAuthn requires (by anti-phishing design) interactive registration in the
browser. `just yubikey` generates the Kanidm reset link to send to the user;
for Vaultwarden, the key is added in the web vault (*Settings → Security →
Two-step login → WebAuthn*).
:::

:::caution[Two pitfalls of the web ceremony]
- **Password manager**: Bitwarden or KeePassXC intercept creation and store the
  passkey in **their vault**, not on the physical key (symptom at login: the key
  blinks then `NotAllowedError`). Decline their popup (« Use the browser ») or
  temporarily disable the « Save passkeys » option in the extension.
- **FIDO2 PIN required**: Kanidm requires user verification, so the browser
  asks for the key's PIN in addition to the touch. Session, sudo and LUKS never
  ask for the PIN: they remain touch-only.
:::

## One key for multiple accounts

The same physical key can serve multiple users: simply repeat enrollment for
each login, key plugged in.

```sh
just yubikey <login1>
just yubikey <login2>   # same physical key, different account
```

Each enrollment creates credentials **specific to the account** (PAM, LUKS, and
a Kanidm reset link per user to redo the web ceremony with the same key).
Kanidm passkeys are non-resident: they do not occupy any slot on the key, the
number of accounts is unlimited. A revocation (`just yubikey <login> main
revoke`) only affects the targeted login.

## Under the hood

| File | Content | Status |
|---|---|---|
| `usr/secrets/yubikeys.json` | Public credentials (PAM, FIDO2 ID, salt) | Public, versioned |
| `secrets.yaml` → `yubikey/<login>/<key>/luks-secret` | LUKS secret derived from the key | Encrypted with sops |
| `secrets.yaml` → `luks-passphrase` | Passphrase entered during disko install | Encrypted with sops (requested on 1st enrollment) |

- **PAM**: the origin is fixed (`pam://<domain>`), so one credential works on
  the entire fleet. The `darkone.system.yubikey` module (enabled automatically
  as soon as the registry exists) distributes `/etc/u2f_mappings` on each host.
- **LUKS**: the derived secret (`HMAC(key, salt)`) can only be computed with
  the physical key. Each encrypted host enrolls it in its LUKS header (keyslot +
  `systemd-fido2` token, the exact format from `systemd-cryptenroll`) via an
  idempotent service, and removes those no longer declared.

:::caution[Fixed origin]
Changing `darkone.system.yubikey.origin` (or the network domain) invalidates
all PAM credentials: all keys would need to be re-enrolled.
:::

## Revoking a lost key

```sh
just yubikey <login> main revoke   # or backup
just commit "user(yubikey): revoke <login>" && just apply
```

Upon deployment, the PAM credential disappears from hosts and the corresponding
LUKS keyslots are removed. Remember to also remove the passkey from Kanidm
(web interface or reset token) and Vaultwarden.

## Limits and future

- TOTP remains available on the web side (Kanidm, Vaultwarden), not at local login.
- LUKS unlock **without PIN code** (touch only) in this version.
- Coming soon: FIDO2 SSH keys (`ed25519-sk`) and git signing, mandatory
  second-factor mode per host, admin sops decryption gated by the key
  (`age-plugin-fido2-hmac`).
