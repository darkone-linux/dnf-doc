A DNF **"service" module** wires a self-hosted piece of software (web UI, API,
storage…) into the framework: reverse proxy, homepage, persistence, firewall,
SSO and per-host activation. This page describes how to create and maintain such
a module, from header to commit, using the `oxicloud` module as a running
example.

<Aside type="note" title="Prerequisites">
First read the service module standard (`dnf/modules/service/AGENTS.md`) and the
[code headers](/en/doc/how-to/dev/code-headers/). Option details are generated in
the [reference](/en/ref/modules/service/).
</Aside>

{/* t:p 76486290b1c9c3abae090963f47c1e53893ce7e224088dc8301735d395d8e227 */}
## Module anatomy

A module lives in `dnf/modules/service/<name>.nix`. It is **auto-imported**
via the generated `default.nix` (run `just generate` after creation).

{/* t:p 488bb3fa8a1cbf8e337613e39af5b6761ea5b5a026b413d9845c0f9c8fec65c8 */}
### Header

Mandatory header comment: 1st descriptive line, then paragraphs and
Starlight admonitions (`note`, `tip`, `caution`, `danger`) oriented toward
usage / maintenance.

```nix
# OxiCloud : Fast Sovereign Cloud (file storage, WebDAV, CalDAV & CardDAV).
#
# :::note[Service currently being validated]
# DNF wrapper around the nixpkgs module (from PR #516113).
# :::
```

{/* t:p 4ea9c63b98475b9a48e345ab4f8add826cc152848fc8767eb3d4682b845c40d5 */}
### Options

Imposed schema: `darkone.service.<name>.<option>`. Start with only the `enable`
option; only add options for matters not resolved by the framework.

```nix
options = {
  darkone.service.oxicloud.enable = lib.mkEnableOption "Enable local OxiCloud service";
};
```

<Aside type="caution" title="Not an option: service parameters">
`domain`, `global`, `title`… are **not** module options: they are parameters
declared in `etc/config.yaml`, resolved by the framework via
`dnfLib.extractServiceParams`. See [Per-host activation](#registration-and-activation).
</Aside>

{/* t:p 39c1e7d40a9083fc070b39d990424dffb66f3673b1a72e29227642a1f3fdec8c */}
### `config` structure

Always `lib.mkMerge` of two blocks:

<Steps>

1. **Unconditional block**: registration with the service system and

2. **`lib.mkIf cfg.enable` block**: the actual NixOS configuration (the upstream
   service, the firewall, the secrets…).

</Steps>

```nix
config = lib.mkMerge [
  {
    darkone.system.services.service.oxicloud = { /* registration */ };
    darkone.service.idm.oauth2.oxicloud = { /* template OAuth2 */ };
  }
  (lib.mkIf cfg.enable {
    darkone.system.services = dnfLib.enableBlock "oxicloud";
    services.oxicloud = { /* … */ };
  })
];
```

Two-block structure:

```d2
style.fill: transparent
direction: down

module: "Module <name>.nix" {
  header: Header
  options: Options
  config: "config =\nlib.mkMerge"
}

incond: "Unconditional block always evaluated" {
  proxy: Reverse proxy
  persist: Persistence
  oauth2: OAuth2
}

cond: "mkIf cfg.enable block (enabled)" {
  service: NixOS service
  firewall: Firewall
  secrets: Secrets
}

config -> incond
config -> cond
```

## Injected arguments and consumer boundary

Beyond `lib`, `config`, `pkgs`, each service module receives `specialArgs`:

- `host`: current host (`hostname`, `zone`, `ip`, `vpnIp`, `services`, `profile`…).
- `hosts`: all hosts in the deployment (all zones).
- `zone`: current host's zone (`name`, `domain`, `gateway`, `lang`…).
- `network`: global config (`domain`, `coordination`, `services`, `zones`).
- `dnfLib`: framework helpers (never import `dnf/lib/` manually).
- `dnfConfig`: framework config registries (`dnfConfig.network.ports.<key>`: port registry).
- `workDir`: consumer workspace root.
- `users`: all users.

<Aside type="danger" title="Framework ↔ consumer boundary">
A framework module accesses consumer files (`usr/secrets/…`,
`usr/www/…`) **only** via `workDir + "/usr/…"`, never by relative path
(`./../../usr/…`). The framework flake root is not the consumer workspace.
</Aside>

## Registration and activation

{/* t:p 47b4195921cbf9e27ef69e887f3ad700d18ef27c28feaec1954ea2b297985727 */}
### Service exposition

The **unconditional** block declares `darkone.system.services.service.<name>`, read
by `modules/system/services.nix` to generate reverse proxy, homepage,
persistence and backup.

```nix
darkone.system.services.service.oxicloud = {
  inherit defaultParams;                 # title / description / icon homepage
  persist = {
    dirs = [ oxCfg.dataDir ];            # service data
    dbDirs = [ config.services.postgresql.dataDir ];
  };
  proxy.servicePort = srvPort;           # Caddy proxies to this port
};
```

- `proxy.servicePort`: internal HTTP port; the Caddy reverse proxy connects to it
- `persist.{dirs,dbDirs,varDirs,mediaDirs,…}`: persisted / backed-up paths.
- `defaultParams`: default metadata (`title`, `description`, `icon`),
  overridable from `config.yaml`. The `icon` uses the
  [selfh.st](https://selfh.st/icons/) slug (prefixed `sh-` by the framework).

```nix
defaultParams = {
  title = "OxiCloud";
  description = "Fast Sovereign Cloud";
  icon = "oxicloud";
};
```

Protocols served on the same HTTP port (WebDAV, CalDAV, CardDAV for
OxiCloud) naturally go through the single Caddy vhost: nothing special to
configure.

### Per-host activation

A service activates on a host when it appears under `services:` in
`etc/config.yaml`, or via a trigger in `dnf/config/modules.nix`:

```nix
oxicloud = {
  # reverseProxy = true by default (web service with UI/API).
  activation.profiles.minimal.triggers.keys.oxicloud = [ "enable" ];
};
```

Available flags:

| Flag | Default | Role |
|---|---|---|
| `reverseProxy` | `true` | Exposes the service via Caddy |
| `uniquePerZone` | `false` | Single instance per zone |
| `externalAccess` | `false` | External access |
| `require` | `[ ]` | Dependencies to activate on the same host (error otherwise) |

On the `config.yaml` side, the following
example installs a full OxiCloud on `https://cloud.mondomaine.tld`:

```yaml
hosts:
  - hostname: "my-server"
    services:
      oxicloud:
        domain: "cloud"
        global: true
```

The framework resolves `domain`/`global` into `params` (`fqdn`, `href`, `ip`…) via
`dnfLib.extractServiceParams host network "oxicloud" defaultParams`.

## Network, database and secrets

{/* t:p 26765002e9ee8af933ac0511563de3dec5e099583b16273a1c080d7af79f5df3 */}
### Internal ports

Any internal port a module **binds**, **opens** in the firewall or **proxies**
must come from the central registry `dnf/config/network.nix`, read via
`dnfConfig.network.ports.<key>` → never a hardcoded literal in the module.
It is the single source of truth: the same number is never written twice and two
services cannot silently conflict.

<Steps>

1. **Name the key** `<service>[Usage]`: the `ports.` prefix already says "port",
   so the key specifies *what it is used for* (`kanidmReplication`, `garageRpc`). A
   `<service>` alone designates the main port. Keep `network.nix` sorted by value.
2. **Add the value** in `ports.<key>` when DNF itself binds the port, then
   read it in the module:

   ```nix
   srvPort = dnfConfig.network.ports.oxicloud;
   ```

3. **Port inherited from upstream** (`config.services.<x>.port`, not bound by DNF):
   register it in `ports.reserved` (anti-conflict denylist) instead; the module
   continues to read `config.services.<x>.port`.

</Steps>

<Aside type="note" title="Tested uniqueness">
`dnf/tests/unit/config/network_test.nix` checks global uniqueness across all
`ports.*` values and the `ports.reserved` list. Any conflict fails the unit
test suite.
</Aside>

{/* t:p ce5f822760b5b07f5a3ea4233f2d59de88b4f575227356dfa7569b909f348bd0 */}
### Internal firewall

Web service behind Caddy → no external opening. Open the port on internal
interfaces only with `dnfLib.mkInternalFirewall`:

```nix
networking.firewall = dnfLib.mkInternalFirewall host zone [ srvPort ];
```

<Aside type="caution" title="TLS outside HCS">
Outside HCS, a `DynamicUser` service cannot read synced Caddy certificates. For
a web service, rely on the automatic reverse proxy (`proxy.servicePort`) + HTTP
in the clear on the trusted LAN/VPN; do not manage ACME manually.
</Aside>

{/* t:p a1ae86ae2dc293a9864a5933ab1e6dc10528bc2bd564a0ea7ff245a8dfaf4d9e */}
### Local PostgreSQL

Many upstream modules expose a flag for local database creation. For OxiCloud,
the option is sufficient (peer auth via socket):

```nix
services.oxicloud.createLocalDatabase = true;   # creates DB + role "oxicloud"
services.postgresqlBackup.enable = true;        # backup
```

Persist the PostgreSQL `dataDir` via `persist.dbDirs` (see above). Without an
upstream option, create DB and role manually (see `matrix.nix`).

{/* t:p cb89811f7cef71425cb6cd6a8f7773224e409bff33101b4987d132cc7b31a541 */}
### Sops secrets

- Declare `sops.secrets.<name> = { mode = "0400"; owner = "<svc-user>"; }`
  (default file `usr/secrets/secrets.yaml`).
- A `DynamicUser = true` service cannot **own** a sops secret. If it loads the
  key via `LoadCredential` (read as root), a root `0400` secret suffices;
  otherwise, fall back to a static group + `SupplementaryGroups`.

<Aside type="tip" title="External binaries in systemd units">
**Every external binary** in a systemd unit (`script`, `ExecStart=`, `preStart`…)
must be written as a full store path: `${pkgs.<pkg>}/bin/<cmd>` (systemd's PATH
is empty). Shell *builtins* (`echo`, `[`, `for`…) are exempt.
</Aside>

## OIDC Authentication with Kanidm

SSO wiring is **automatic and conditional** on the presence of an `idm`
(Kanidm) service on the network (zone or global). Three ingredients:

<Steps>

1. **OAuth2 Template** (unconditional block): `darkone.service.idm.oauth2.<name>`.
   Kanidm provisions it as an OAuth2 client when `idm` is enabled.
2. **OIDC Context**: `dnfLib.mkOidcContext` → `clientId`, `secret`, `idmUrl`.
   `idmUrl == null` when no `idm` exists: all wiring is then cut off.
3. **Endpoints**: `dnfLib.mkKanidmEndpoints idmUrl clientId` → `issuerUrl`,
   `authUrl`, `tokenUrl`, `userinfoUrl`…

</Steps>

```nix
params = dnfLib.extractServiceParams host network "oxicloud" defaultParams;
inherit (dnfLib.mkOidcContext {
  name = "oxicloud";
  inherit params network hosts;
}) clientId secret idmUrl;
oidc = dnfLib.mkKanidmEndpoints idmUrl clientId;
hasIdm = idmUrl != null;
```

The template declares **redirect paths** (prefixed by the service's public URL
framework) and a landing page:

```nix
darkone.service.idm.oauth2.oxicloud = {
  displayName = "OxiCloud";
  imageFile = ./../../assets/app-icons/oxicloud.svg;
  redirectPaths = [ "/api/auth/oidc/callback" ];
  landingPath = "/";
};
```

The OAuth2 secret is generated on the Kanidm side (`oidc-secret-<clientId>`). To
provide it to the service without storing it in the Nix store, **re-encrypt** it
for the service user (sops `key` field) then inject it via an environment file
(`sops.templates`, `KEY=value` format):

```nix
sops.secrets."${secret}-service" = lib.mkIf hasIdm {
  mode = "0400";
  owner = "oxicloud";
  key = secret;                          # master secret alias
};

sops.templates."oxicloud-oidc-env" = lib.mkIf hasIdm {
  content = "OXICLOUD_OIDC_CLIENT_SECRET=${config.sops.placeholder."${secret}-service"}";
  mode = "0400";
  owner = "oxicloud";
  restartUnits = [ "oxicloud.service" ];
};

services.oxicloud = {
  settings.oidc = lib.mkIf hasIdm {
    enable = true;
    issuerUrl = oidc.issuerUrl;
    inherit clientId;
    redirectUri = "${params.href}/api/auth/oidc/callback";
    frontendUrl = params.href;
  };
  environmentFiles = lib.mkIf hasIdm [ config.sops.templates."oxicloud-oidc-env".path ];
};
```

<Aside type="tip" title="Conditionality">
The OAuth2 template remains declared unconditionally (harmless without `idm`).
Only the service-side wiring (`settings.oidc`, secret, env) is guarded by
`lib.mkIf hasIdm`, leaving upstream behavior intact in the absence of Kanidm.
Reference patterns: `outline.nix`, `immich.nix`, `forgejo.nix`.
</Aside>

## Protecting a service without OIDC

When the service **cannot** speak OIDC (landing page, static site, UI without
auth), DNF protects its vhost upstream via **oauth2-proxy** + **Caddy forward
auth**, backed by Kanidm. A single oauth2-proxy instance per gateway, shared
by all protected services in the zone. Everything is declarative on the service
record (unconditional block), via two `proxy` options:

```nix
darkone.system.services.service.oxicloud = {
  inherit defaultParams;
  proxy.servicePort = srvPort;
  proxy.isProtected = true;            # places the vhost behind oauth2-proxy
  proxy.allowedGroups = [ "users" ];   # allowed groups (SPN added by the framework)
};
```

| Option | Default | Role |
|---|---|---|
| `proxy.isProtected` | `false` | Puts the vhost behind oauth2-proxy (forward auth) |
| `proxy.allowedGroups` | `[ ]` | Allowed Kanidm groups (empty: any authenticated user) |

<Aside type="note" title="Anchor and prerequisites">
The login flow (`/oauth2/*`) is hosted on the zone's landing page FQDN: the
`homepage` service must therefore be present as soon as a service is protected
(an assertion checks this). Groups are compared by **SPN**
(`name@<domain>`) : the framework appends the suffix, declaring the short name
suffices.
</Aside>

<Aside type="tip" title="Native OIDC or forward auth?">
If the service **can** speak OIDC, prefer native wiring (previous section): it
passes identity and roles to the service. Forward auth only **guards the door**
in front of a service that ignores who connects. Admin overview:
[Authentication and identities](/en/doc/admin-guide/understand/auth-idm/).
</Aside>

## Sourcing an upstream module

When only the NixOS **module** is missing (package already in nixpkgs), source it
from a fork carrying the PR, without rebuilding it.

<Steps>

1. Add in `dnf/flake.nix` a dedicated input pointing to the fork branch,
   **without** `inputs.nixpkgs.follows` (we want the fork tree as-is):

   ```nix
   nixpkgs-oxicloud.url = "github:flashonfire/nixpkgs/oxicloud-service";
   ```

2. Inject the module path into the `modules` list of `mkNode`
   (`dnf/lib/mk-configuration.nix`) → parsed unconditionally, with no effect
   as long as `services.<name>.enable = false`:

   ```nix
   "${nixpkgs-oxicloud}/nixos/modules/services/web-apps/oxicloud.nix"
   ```

</Steps>

<Aside type="caution" title="Package overlay: only if needed">
Distinguish two cases. **OxiCloud**: the package is already in nixpkgs → no
overlay, `pkgs.oxicloud` comes from the main tree. **GeneWeb**: the package was
missing (OCaml deps) → an overlay (`lib/overlays/geneweb.nix`) injects
`pkgs.geneweb` from the fork.
</Aside>

<Aside type="tip" title="Cleanup after merge">
**Cleanup** once the PR is merged into `nixos-unstable`: remove the input,
remove the module path (and any overlay). The wrapper
`modules/service/<name>.nix` stays unchanged.
</Aside>

## Finalize and commit

<Steps>

1. `just generate`: regenerates `default.nix` (auto-import of the new module).
2. `just clean`: fix + check + generate + format (mandatory before commit).
3. Evaluate / build a host to validate, then commit.

</Steps>

Commit message: one line, 80 characters max, English, format
`<type>(<subject>): <message>`, e.g. `module(oxicloud): full DNF service with OIDC`.
